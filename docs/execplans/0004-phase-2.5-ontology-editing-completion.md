# ExecPlan: Phase 2.5 User-Facing Ontology Editing Completion

## Status

Completed

## Goal

Complete the remaining basic user-facing ontology editing operations through the existing Kotlin semantic engine, machine-readable CLI, and VS Code workbench without creating a second editing pathway.

Every edit must reuse the Phase 2 proposal lifecycle: typed edit, graph change translation, preview, semantic diff, deterministic validation, Turtle round-trip verification, explicit approval or rejection, atomic application, reload, and rollback.

## Related Spec

- `docs/specs/0004-phase-2.5-ontology-editing-completion.md`
- `docs/architecture/phase-2.5-scope.md`

## Current State

- Phase 1, Phase 1.5, and Phase 2 are merged.
- `core-types` already contains typed-edit contracts for classes, properties, individuals, assertions, superclass relationships, domains, ranges, and labels.
- `semantic-engine` already translates the typed contracts and owns proposal preview, baselines, Turtle round-trip verification, atomic application, reload, and rollback.
- `validation-engine` already validates proposal structure, preview graphs, baselines, and semantic equivalence, but needs edit-specific compatibility checks.
- `graph-diff` already generates proposal diffs.
- At the start of Phase 2.5, `cli` exposed the proposal lifecycle but its user-facing edit options supported only `create-class`.
- `vscode-extension` currently exposes a focused `create-class` form and shared proposal result flow.
- Existing tests cover Phase 1, Phase 1.5, Phase 2 behavior, and the `create-class` end-to-end path.

## Target State

- The machine-readable CLI accepts every Phase 2.5 edit kind through a stable extension of the existing proposal commands.
- The CLI translates requests into existing typed-edit contracts and returns consistent proposal, validation, diff, apply, reject, and rollback payloads.
- Deterministic validation handles edit-specific references, duplicate changes, missing removals, literal compatibility, and known domain/range compatibility.
- The VS Code workbench provides focused form modes for every Phase 2.5 edit kind while reusing one preview and approval flow.
- Copied-fixture end-to-end tests prove that successful, rejected, stale, and rollback paths preserve the Phase 2 safety guarantees.
- No new modules, semantic-web frameworks, server infrastructure, or Git automation are introduced.

## Dependency Order

Implement the slices serially in this order:

1. CLI edit request and dispatch foundation.
2. Deterministic edit-specific validation.
3. CLI property, domain, and range flows.
4. CLI individual and assertion flows.
5. CLI superclass and label flows.
6. Shared VS Code edit selector and form-state boundary.
7. VS Code property, domain, and range forms.
8. VS Code individual and assertion forms.
9. VS Code superclass and label forms.
10. End-to-end Phase 2.5 regression.

Slices 3, 4, and 5 should not be implemented in parallel because they share CLI option parsing and proposal preparation files. Slices 7, 8, and 9 should not be implemented in parallel because they share the webview form and message boundary. Slice 10 must run after all behavior slices.

After slices 1 and 2 establish stable request and validation contracts, test-only work may be parallelized if each agent owns separate test files. Do not parallelize edits to `core-types`, `shared`, build files, specs, ExecPlans, `AGENTS.md`, shared CLI JSON contracts, or shared workbench message models.

## Implementation Slices

### Slice 1: CLI Edit Request And Dispatch Foundation

#### Goal

Extend the existing proposal command boundary so it can parse all Phase 2.5 edit kinds and construct the corresponding existing typed-edit contracts without duplicating semantic logic.

#### Allowed files/modules

- `cli/src/main/kotlin/com/entio/cli/ProposalCommands.kt`
- `cli/src/main/kotlin/com/entio/cli/ProposalCommandSupport.kt`
- `cli/src/main/kotlin/com/entio/cli/JsonOutput.kt`
- `cli/src/test/kotlin/com/entio/cli/`
- `semantic-engine/src/main/kotlin/com/entio/semantic/TypedOntologyEditTranslator.kt` only for a minimal compatibility correction required to dispatch an already defined contract
- Matching semantic-engine tests if that correction is required
- `docs/decisions/` completion artifact

#### Forbidden actions/modules

- Do not change `core-types` contracts unless a separately approved contract correction is created.
- Do not add a JSON parser, server, dependency, or new CLI command family.
- Do not implement validation policy, UI forms, source writes, or Git automation in this slice.
- Do not add edit behavior that is absent from the existing typed-edit model.

#### Expected changes or output

- A typed internal request model maps edit kind and options to existing typed-edit objects.
- Existing `create-class` arguments remain backward compatible.
- Unsupported or incomplete requests return stable structured issues.
- Proposal commands can dispatch every Phase 2.5 edit kind to the shared proposal preparation path.

#### Tests

- Argument parsing tests for every edit kind.
- Backward compatibility tests for `create-class`.
- Unsupported, missing-field, and malformed-input tests.
- Typed request construction tests.

#### Verification commands

```bash
./gradlew :cli:test
./gradlew test
```

#### Stop conditions

- Dispatch requires new core contracts not approved by the spec.
- A new dependency or structured-input protocol is required.
- CLI parsing begins to implement RDF or validation semantics directly.

- Checks for existing resources, expected symbol kinds, duplicate additions, missing removals, datatype values, known domains/ranges, self-subclass relationships, and duplicate superclass relationships.

#### Tests

- Valid and invalid tests for each new validation category.
- Deterministic ordering tests.

## Rollback Notes

- Revert later CLI slices before reverting the shared CLI request foundation.
- Revert workbench form slices independently; they should not require changes to Kotlin persistence behavior.
- Revert validation changes only after dependent CLI and workbench tests are removed or adjusted.
- Keep fixture changes temporary and reversible.
- Preserve the Phase 2 apply and rollback implementation while reverting edit-surface slices.
- Do not use destructive Git commands to discard user work.

## Risks And Assumptions

Risks:

- A growing set of picocli options could make the machine-readable boundary difficult to maintain.
- Workbench forms could duplicate semantic validation instead of delegating to Kotlin.
- Domain and range compatibility may be underspecified without reasoning.
- Literal lexical validation may differ across supported datatypes.
- Label replacement and multi-valued properties may require composed graph changes.
- Multiple ontology sources may make entity selection and target-source choice ambiguous.
- The workbench could expand into a full graph editor if form scope is not constrained.

Assumptions:

- Existing `core-types` typed-edit contracts are sufficient for the required operations.
- Apache Jena remains the RDF/Turtle serialization and semantic-equivalence library.
- Phase 2 proposal state remains in memory.
- Phase 2.5 continues to target small local Turtle projects.
- No new dependencies are required for the initial implementation.
- External Git workflows remain outside Entio.

## Definition Of Done

Phase 2.5 is complete when:

- The spec-defined edit kinds work through the machine-readable CLI.
- The VS Code workbench exposes a focused form for each edit kind.
- All edits use the existing proposal preview, diff, validation, approval, apply, reload, and rollback path.
- Invalid, duplicate, missing, incompatible, stale, rejected, and failed-apply cases are covered.
- Only intended ontology sources change after successful application.
- Temporary and copied-fixture end-to-end tests pass.
- `./gradlew test`, `./gradlew build`, `./gradlew check`, and `npm test` pass.
- No Phase 2 non-goal or new standards framework was introduced.
- A completion summary documents actual implementation and remaining limitations.

## Boundary Check

This plan fits the current active project direction by completing the user-facing ontology editing surface that Phase 2 intentionally left partial.

It preserves module responsibilities, keeps semantic behavior in Kotlin, keeps TypeScript thin, uses existing RDF/Turtle tooling, avoids new infrastructure, and excludes all Phase 2 and Phase 2.5 non-goals.
