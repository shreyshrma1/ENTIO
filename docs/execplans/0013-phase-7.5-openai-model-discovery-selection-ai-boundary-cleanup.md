# ExecPlan: Phase 7.5 OpenAI Model Discovery, Selection, And AI Boundary Cleanup

## Status

Approved and implementation-ready after the Planning Boundary Prerequisite is satisfied. It does not activate Phase 7.5 implementation by itself.

## Goal

Replace Entio's single pinned OpenAI model requirement with a server-side workflow that discovers models visible to each user's API key, filters them through an Entio-owned compatibility policy, requires explicit selection and bounded verification, and binds future AI runs to the verified model. Reorganize and document the AI implementation tree only as needed to keep provider, credential, model, capability, conversation, draft, audit, and web responsibilities clear.

Kotlin remains authoritative for credentials, eligibility, selection, run binding, semantic capabilities, drafts, deterministic analysis, and review handoff. React remains a versioned client and never receives credentials, raw provider inventory, or compatibility policy.

## Fixed Implementation Decisions

### Provider endpoints and authority

- Production discovery calls `GET https://api.openai.com/v1/models` from `web-server` only.
- Production inference remains `POST https://api.openai.com/v1/responses` with server-enforced storage, tool, timeout, and retry policy.
- Provider redirects to non-approved hosts are rejected.
- OpenAI remains authoritative for project/model permissions and provider-side rate limits.
- Entio does not attempt to infer or replicate the user's actual OpenAI quota.

### Dynamic model eligibility

- Entio does not require a fixed exact model allowlist.
- A model is eligible for user consideration only when:
  1. it is returned by `/v1/models` for the current user's credential;
  2. it is not excluded by Entio's unsupported-category policy;
  3. the user explicitly selects it;
  4. it passes Entio's live compatibility verification.
- Free-text model IDs are not accepted.
- A client may select only an ID contained in the current server-returned candidate set for that user and credential generation.
- Unknown-but-discovered models may be considered candidates.
- Unknown models do not become usable until verification succeeds.
- Entio may provide richer display metadata for known models, but metadata is optional and does not determine availability.
- Moving aliases such as IDs ending in `latest` are rejected unless separately approved later.
- Obviously non-conversational categories are excluded from the selector, including embedding, moderation, image-generation, transcription, speech, and other provider resources that cannot support the Phase 7 tool loop.
- The unsupported-category filter is server-owned and tested; React never reproduces it.

### Selection and verification

- One eligible candidate still requires explicit user selection.
- Selected-model verification uses a minimal harmless Entio-owned custom function and no project context.
- Verification may incur a small charge and the UI discloses that before the call.
- A model is marked usable only after successful Responses API and custom-tool verification.
- No automatic fallback or silent model switching is allowed.
- Selection and verification are per user, in memory, and tied to a credential generation.
- Replacing or removing a credential clears discovery and selection.
- Runs capture the exact selected model ID and current Entio request policy at creation; no run switches model.

### Discovery freshness

- Discovery results remain fresh for 15 minutes.
- Explicit `Refresh models` always bypasses the cache and performs a provider call.
- No deployment-level freshness override is added in Phase 7.5.

### Entio local protection limits

These limits are local safeguards against accidental repeated calls. They do not represent OpenAI quotas.

- Model discovery: at most 5 provider discovery calls per user and credential generation in any rolling 15-minute window.
- Model verification: at most 3 verification calls per user and credential generation in any rolling 15-minute window.
- Concurrent discovery: at most 1 per user and credential generation.
- Concurrent verification: at most 1 per user and credential generation.
- Provider HTTP 429 responses remain authoritative and are mapped to structured rate-limit states.
- Safe provider rate-limit headers may be used for user-facing retry guidance without exposing raw headers.
- Replacing the credential creates a new credential generation but must not be used to bypass server abuse controls; a small server-wide user-session guard may remain in effect.

### Compatibility migration

- Existing credential-status and credential-test routes remain available throughout Phase 7.5.
- They may be marked deprecated internally but are not removed in this phase.
- New React code must use the new unified provider/model status by the end of Slice 6.
- Existing deterministic providers gain deterministic discovery/verification fixtures and remain the CI default.

### Package cleanup compatibility

- Use direct import updates for package-private and `web-server`-internal types.
- Use temporary forwarding aliases or compatibility wrappers only when a moved type is referenced outside `web-server`, through an existing documented public contract, or by tests representing supported external behavior.

Automated verification must never require a live API key or external network. A manual provider smoke test is optional, may incur cost, and is not a completion artifact.

## Rollback Notes

- Each slice should be one focused commit or a small reviewable series so it can be reverted independently.
- Slice 1 file moves are behavior-preserving and can be reverted without touching later semantic modules.
- Compatibility, discovery, selection, contracts, and UI remain additive until runtime binding. Reverting the complete dependent Slice 2 through Slice 7 chain restores prior pinned-model behavior; reverting only a middle slice may require reverting its dependent later slices.
- Runtime binding must retain a compatibility seam until the complete Phase 7.5 regression passes. Reverting Slice 7 restores the prior pinned provider configuration.
- New routes can be removed after reverting React usage; legacy credential routes remain during migration.
- In-memory settings require no durable migration or data rollback.
- Never roll back by weakening credential redaction, cross-user isolation, human review, or semantic validation.

## Risks And Assumptions

- OpenAI model availability and IDs change over time. Availability comes from the user's live provider inventory; Entio uses a stable category filter plus live verification rather than a fixed exact allowlist.
- `GET /v1/models` proves visibility, not tool compatibility; verification remains mandatory.
- Verification can incur a small provider charge and must be idempotent and disclosed.
- A flat-tree cleanup can create noisy diffs; the architecture map must justify moves and permit no-op cleanup.
- Existing credential status contracts may be consumed by tests or clients not obvious from the settings UI; additive migration and repository-wide search are required.
- Development identity does not provide production session expiration; Phase 7.5 must not claim production credential/session security.
- Provider error categories may not be perfectly stable; mapping should use HTTP status and known safe codes while redacting raw messages.
- The 15-minute freshness interval balances provider calls and access changes but may need later operational tuning.
- Per-user abuse limits are in-memory development controls, not a distributed production rate limiter.
- The approved base must resolve any existing regression failures before Phase 7.5 completion can be claimed.

## Resolved Implementation Decisions

The following decisions are fixed for Phase 7.5:

- No fixed exact model allowlist is required.
- Candidate models must come from the current user's live `/v1/models` response.
- Entio filters clearly unsupported resource/model categories on the server.
- Unknown-but-discovered candidates may be selected and verified.
- Free-text or undiscovered model IDs are rejected.
- Live Entio verification determines usability.
- Provider limits remain authoritative to OpenAI.
- Entio local discovery limit is 5 calls per user and credential generation per rolling 15 minutes.
- Entio local verification limit is 3 calls per user and credential generation per rolling 15 minutes.
- One concurrent discovery and one concurrent verification are allowed per user and credential generation.
- Existing credential-status and credential-test routes remain available for the full phase.
- Direct import updates are used for internal package moves; compatibility aliases are reserved for externally referenced or documented contracts.
- Discovery freshness is fixed at 15 minutes.
- Explicit refresh bypasses the cache.
- No deployment-level freshness override is added in this phase.

No product or architectural decision remains open before implementation.

## Definition Of Done

Phase 7.5 is complete only when:

- All nine slices are implemented serially and reviewed.
- AI subsystem ownership is documented and cleanup is demonstrably behavior-preserving.
- The Entio compatibility policy, live provider inventory, explicit user selection, and live verification deterministically control usability.
- Model discovery uses the current user's server-side credential and exposes no raw inventory.
- Explicit selection and minimal verification are required before ready state.
- Selection is isolated per user, tied to credential generation, and cleared appropriately.
- Every AI run uses one captured model for its lifetime.
- Model access loss preserves drafts/conversations and requires explicit reselection without fallback.
- React renders accessible lifecycle states and never retains the credential.
- Credentials, headers, raw payloads, and unrestricted model inventory are absent from all external/state boundaries.
- Non-AI functionality remains available through all AI failure states.
- ADR, architecture map, catalog documentation, help, and Phase 7.5 summary match verified behavior.
- Focused and full Kotlin, React, Playwright, and VS Code verification pass.
- No semantic module, ontology configuration, typed-edit policy, proposal authority, or human-review behavior changes.

## Boundary Check

- Phase fit: the plan is an additive Phase 7 provider/credential refinement within Phase 7.5 scope.
- Current non-goals: it avoids persistence, billing administration, arbitrary routing, unrestricted model IDs, multi-provider UI, document ingestion, and autonomous authority.
- Module ownership: provider/session/web behavior stays in `web-server`; UI stays in `web-app`; semantic modules are forbidden edit targets.
- Dependency direction: no engine module depends on web/provider/client code, and `shared` receives no product logic.
- Standards: no RDF, OWL, SHACL, parsing, reasoning, or validation framework is added or replaced.
- Speculative infrastructure: no new module, database, queue, provider SDK, pricing service, or broad framework is introduced.
