# ExecPlan: Phase 10.5 Inferred Facts In Explore And Ontology Map

## Status

Approved for implementation on 2026-07-23.

- [Phase 10.5 scope](../architecture/phase-10.5-scope.md)

Slice 0 records the approved contract audit before production implementation begins.

## Goal

Expose supported inferred facts from both the current applied graph and the current proposal graph throughout Explore and the ontology map.

Users receive two independent read-only visibility controls:

- `Show inferred for applied`
- `Show inferred for proposal`

Entio automatically keeps project reasoning current whenever applied graph state or proposal graph state changes. All project views reuse those current project-owned reasoning results instead of running reasoning separately.

The delivery path is:

```text
applied or proposal graph changes
→ Entio reruns reasoning for that graph state
→ project-owned current reasoning result is updated
→ authorized Explore and map reads reuse that result
→ applied and proposal inference controls decide what is visible
→ inferred facts appear in their normal fields
→ inferred map relationships appear with clear inferred styling
```

No slice stages, materializes, applies, or writes an inferred fact.

## Approved Product Decisions

The implementation must preserve these decisions:

- Reasoning results belong to the project, not to the user who started reasoning.
- All authorized users viewing the same project reuse the same current reasoning results.
- Entio maintains one current applied-graph reasoning result and one current proposal-graph reasoning result.
- Applied reasoning reruns when applied project graph state changes, including load, apply, reload, and rollback.
- Proposal reasoning reruns when proposal graph state changes, including staged edit add, update, remove, rebuild, and clear.
- Project views must reuse current project reasoning results rather than run reasoning independently.
- Explore provides two temporary read-only controls:
  - `Show inferred for applied`
  - `Show inferred for proposal`
- Both controls are off by default.
- The controls are shared across the current project's open Explore tabs and are not persisted across reload or project changes.
- `Reset filters` turns both controls off.
- Enabling either control does not itself start a separate reasoning job.
- When reasoning for the selected graph state is still running, asserted content remains visible and the UI shows that inferred facts are updating.
- When reasoning fails, asserted content remains visible and the UI shows that inferred facts are unavailable for that graph state.
- Supported inferred facts appear in the same existing fields as equivalent asserted facts.
- Supported visibility includes all deterministic relationships already produced by Entio's reasoning and semantic layer between existing ontology entities, including:
  - superclass and subclass relationships;
  - individual types;
  - object-property assertions;
  - effective property domains and ranges where deterministically supported.
- Reasoning never creates a new class, property, or individual.
- Inferred relationships never replace asserted placement or asserted values.
- Asserted and inferred duplicates render once as asserted.
- Existing map and read limits are aggregate asserted-plus-inferred limits, with asserted content taking priority.
- Asserted class hierarchy alone controls primary map layout.
- Inferred edges reuse existing focus/highlight interaction and do not add editing or a new edge-details workflow.
- Existing reasoning explanations are linked only when already supported for the exact fact.
- Large inferred result sets remain fully discoverable through bounded pagination or `Load more`; the browser does not load every inferred fact at once.

## Current State

- `ReasoningService` and semantic-job orchestration produce bounded results with asserted/inferred origin and graph fingerprints.
- Phase 10 provides canonical inference identities and controlled materialization from the Reasoning workspace.
- `ReadOnlyProjectAdapters` supplies project summaries, outlines, hierarchy, search, and entity details.
- `OntologyGraphService` produces asserted nodes and `SubclassOf`, `Domain`, `Range`, `Type`, and `ObjectAssertion` edges.
- `OntologyGraphWebService` provides fingerprint-bound initial and neighborhood graph pages.
- `ProjectWorkspace` owns Explore navigation, Project Outline filters, tabs, and temporary map state.
- Applied and proposal reasoning are not yet exposed as two coordinated project-wide visibility sources in Explore.
- `EntityDetails`, `HierarchyNode`, and related workbench components render asserted semantic fields.
- `OntologyMapShell`, `OntologyGraphRenderer`, `graphMerge`, and `graphLayout` own bounded graph presentation and interaction.
- `OntologyGraphProvenance` currently supports asserted graph edges only.
- The Project Outline and map do not currently consume retained reasoning output.
- Existing map limits are 75 nodes/150 edges initially, 50/100 per expansion, and 300/600 for one open tab.

## Target State

- Neutral core contracts describe inferred read provenance, graph-state origin, availability, freshness, and graph edge provenance additively.
- One project-owned current applied reasoning result and one project-owned current proposal reasoning result are maintained automatically as their graph states change.
- A focused semantic service projects supported inferred facts into existing Explore fields and graph relationships.
- Read adapters and graph services accept explicit applied/proposal inference inputs without owning job or UI state.
- A server coordinator supplies bounded current applied and proposal inference projections to existing read routes.
- Existing read and graph responses include optional applied and proposal inferred facts with explicit source-state metadata.
- Explore exposes two shared temporary controls:
  - `Show inferred for applied`
  - `Show inferred for proposal`
- Classes, properties, individuals, hierarchy rows, and relationship summaries render inferred facts in their corresponding existing fields.
- Inferred entries use light blue plus visible `Inferred` and `Applied` or `Proposal` labeling.
- The map layers clearly marked applied and proposal inferred edges over the asserted hierarchy layout.
- Applied and proposal inferred edges are visually distinct from asserted edges and from each other.
- Existing reasoning materialization and proposal workflows remain unchanged.

## Risks And Assumptions

- **Reasoning lifetime:** current applied and proposal reasoning results are in-memory. They need not survive restart; Entio reruns reasoning as project state is loaded or changed.
- **Project ownership:** reasoning results belong to the project and are shared by authorized project users. User identity still controls project access.
- **Contract concentration:** `ReadOnlyProjectAdapters.kt`, `Application.kt`, and `ProjectWorkspace.tsx` are dense. Focused service/components should contain new behavior without broad refactoring.
- **Duplicate semantics:** canonical triple equality, not label equality, is authoritative.
- **Hierarchy clarity:** inferred multi-parent placement can add rows. Marking, deduplication, cycle guards, and asserted-first ordering must prevent confusion.
- **Layout stability:** inferred edges must be filtered out before primary layout facts are computed; styling alone is insufficient.
- **Bounded endpoints:** a supported inferred edge whose endpoint is outside remaining capacity is omitted until continuation/load-more rather than partially rendered.
- **Staleness:** every inferred page depends on the applied fingerprint; cached frontend data must not outlive that precondition.
- **External/import scope:** inferred facts may depend on imports, but visible endpoints still obey existing browse scope. No imported source becomes writable or newly exposed.
- **Automatic reasoning cost:** applied and proposal graph changes trigger reasoning refresh. Superseded runs must be cancelled or ignored so rapid changes do not create inconsistent results.

## Boundary Check

- **Phase fit:** every slice implements the approved Phase 10.5 read-only visibility scope.
- **Non-goals:** no materialization, editing, AI generation, inference expansion, persistence, CLI, or VS Code capability is added.
- **Infrastructure:** current core contracts, reasoning results, Ktor reads, React Query, Project Outline filters, and map services are extended; no speculative subsystem is introduced.
- **Responsibility:** Kotlin owns semantic facts and provenance, Ktor owns authorized bounded reads, and React owns temporary display state.
- **Dependency direction:** core contracts do not depend on engines; engines do not depend on web modules; web clients do not own semantic policy.
- **Standards:** existing OWL API/HermiT/Jena-backed behavior is reused. No RDF, OWL, or SHACL framework is reimplemented.
