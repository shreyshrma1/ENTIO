# Feature Spec: Phase 2.5 User-Facing Ontology Editing Completion

## Status

Implemented

## Problem

Phase 2 implemented the controlled proposal workflow, graph previews, semantic diffs, deterministic validation, Turtle round-trip verification, atomic application, rollback, a machine-readable CLI boundary, and a minimal VS Code workbench.

The current user-facing workbench demonstrates only one complete edit: creating a class with an optional label. The Kotlin core already contains typed-edit contracts for more ontology operations, but those operations are not consistently available through the CLI and VS Code workbench. Users therefore cannot yet perform the basic property, individual, relationship, value, hierarchy, domain, range, and label edits that Phase 2 intended to support.

Phase 2.5 completes that user-facing editing surface without creating a second mutation pathway. Every new edit must reuse the existing proposal, preview, diff, validation, approval, atomic-save, reload, and rollback workflow.

## Goals

- Expose the remaining basic typed ontology edits through the machine-readable CLI.
- Expose the same edit operations through focused VS Code workbench forms.
- Reuse the existing Kotlin typed-edit contracts and translator.
- Reuse the existing proposal preparation, preview, semantic diff, validation, round-trip verification, apply, reject, refresh, and source-opening behavior.
- Support deterministic validation for required fields, referenced entities, duplicate changes, missing removals, compatible property values, and stale baselines.
- Keep the CLI thin and keep semantic behavior in Kotlin engine modules.
- Keep the VS Code layer responsible for user input, form state, process calls, and result display only.
- Add focused tests for each completed CLI and workbench edit flow.
- Add copied-fixture end-to-end tests that never modify committed examples in place.

## Non-Goals

Phase 2.5 does not include:

- New AI or LLM behavior.
- Schema RAG, document ingestion, document-to-KG conversion, entity resolution, or autonomous agents.
- Full OWL class-expression editing.
- Equivalent-class, disjoint-class, inverse-property, property-chain, cardinality, or property-characteristic editing.
- Full OWL reasoning or inference explanation.
- Full SHACL authoring or validation.
- Source-text-preserving Turtle editing.
- Long-term proposal persistence or project version history.
- Actual Git staging, commits, pushes, branches, or pull requests inside Entio.
- Multi-user collaboration, authentication, authorization, a server, a database, or a separate desktop application.
- A full drag-and-drop ontology graph editor.
- New Gradle modules, a web application, or a new semantic-web framework.

The current Phase 2 non-goals in `AGENTS.md` remain in force.

## Proposed Behavior

Phase 2.5 extends the existing workflow for every supported edit:

1. The user selects an edit type in the CLI or VS Code workbench.
2. The interface collects the edit-specific fields and target ontology source.
3. The Kotlin boundary translates the request into an existing typed ontology edit.
4. The semantic engine translates that edit into one or more graph changes.
5. Entio creates a proposal against the current project baseline.
6. Entio applies the change set to an in-memory preview graph.
7. Entio generates a semantic diff.
8. Entio validates the preview and proposal baseline deterministically.
9. Entio serializes the preview graph to temporary Turtle, reparses it, and verifies semantic equivalence.
10. The user reviews the target source, diff, validation, equivalence result, and affected files.
11. The user rejects the proposal without changing project files, or approves it.
12. Entio rechecks the baseline and applies only an approved, valid, current proposal atomically.
13. Entio reloads the project and verifies the saved graph.
14. Entio restores the prior source and reports rollback status if save verification fails.
15. The workbench refreshes after successful application and can open the changed source.

The current `create-class` flow remains supported. New edit forms must use the same proposal preview and action components rather than bypassing them.

### CLI Boundary

Phase 2.5 should extend the existing `proposal-preview`, `proposal-validate`, `proposal-diff`, `proposal-apply`, and `proposal-reject` commands. The initial design uses the existing picocli process boundary with an `--edit` value and edit-specific options rather than adding a JSON parser dependency.

Common parameters remain:

```text
COMMAND PROJECT_ROOT TARGET_SOURCE_ID --edit EDIT_KIND
```

The boundary should support edit kinds and fields equivalent to:

- `create-class`: `--class-iri`, optional `--label`.
- `create-object-property`: `--property-iri`, optional `--label`, `--domain-iri`, and `--range-iri`.
- `create-datatype-property`: `--property-iri`, optional `--label`, `--domain-iri`, and `--datatype` or `--range-iri`.
- `set-property-domain`: `--property-iri`, `--domain-iri`.
- `set-property-range`: `--property-iri`, `--range-iri` or `--datatype`.
- `create-individual`: `--individual-iri`, optional `--label`, and optional `--type-iri`.
- `assign-individual-type`: `--individual-iri`, `--type-iri`.
- `add-object-property-assertion`: `--subject-iri`, `--property-iri`, `--object-iri`.
- `add-datatype-property-assertion`: `--subject-iri`, `--property-iri`, `--value`, optional `--datatype`, and optional `--language`.
- `add-superclass`: `--class-iri`, `--superclass-iri`.
- `remove-superclass`: `--class-iri`, `--superclass-iri`.
- `set-entity-label`: `--entity-iri`, `--label`, optional `--language`.

The exact option names may be refined during the ExecPlan as long as the boundary remains stable, explicit, machine-readable, and compatible with the existing `create-class` command.

Every command should return the existing structured proposal shape, including proposal status, target source, affected files, preview count, semantic diff, validation report, semantic equivalence, and apply or rollback result where applicable.

### VS Code Workbench

The workbench should add an edit-type selector and focused forms. Forms should reuse the current target-source selector, preview display, validation display, action buttons, refresh behavior, and open-source action.

The workbench should support:

- Creating object properties with optional label, domain, and range.
- Creating datatype properties with optional label, domain, and a small supported datatype range.
- Setting or changing a property domain.
- Setting or changing an object-property class range or datatype-property datatype range.
- Creating an individual with an optional label and at least one optional type assignment.
- Assigning an existing class type to an existing individual.
- Adding an object-property assertion between existing resources or individuals.
- Adding a datatype-property assertion with a literal value and optional datatype or language tag.
- Adding and removing direct superclass relationships.
- Adding or replacing an entity label where supported.

The workbench should disable or reject incompatible selections before submission where practical, but the Kotlin validation result remains authoritative.

## Inputs

Phase 2.5 continues to use small local Entio projects:

```text
project-root/
  entio.yaml
  ontology/
    simple.ttl
```

Inputs include:

- A project root and valid `entio.yaml`.
- One selected ontology source.
- An edit kind and edit-specific fields.
- Existing classes, properties, individuals, and resources when an edit references them.
- Optional labels, language tags, literal values, and supported datatype IRIs.
- Explicit approval or rejection.

The initial supported datatype set is intentionally small:

- `xsd:string`
- `xsd:boolean`
- `xsd:integer`
- `xsd:decimal`
- `xsd:date`
- `xsd:dateTime`

## Outputs

Phase 2.5 produces the existing structured outputs plus edit-specific request and display data:

- Prepared change proposals.
- Preview graphs and graph triple counts.
- Semantic diffs.
- Deterministically ordered validation reports.
- Semantic-equivalence results.
- Applied, rejected, stale, failed, and rolled-back results.
- Changed-file and rollback metadata.
- Workbench form state, selected entities, and machine-readable response models.

The outputs must remain consumable by both the CLI and VS Code extension without exposing Apache Jena types outside the semantic engine boundary.

## Validation Behavior

Validation should preserve all Phase 2 checks and add deterministic checks for the new edit kinds:

- Required fields are present.
- IRIs are nonblank and valid for the supported input boundary.
- The target ontology source exists.
- Referenced classes, properties, individuals, or resources exist when the edit requires existing entities.
- The selected resource has the expected kind where the graph provides that information.
- Duplicate additions are rejected.
- Removals of missing statements are rejected.
- Datatype values can be represented by the selected supported datatype.
- Language tags are accepted only where the RDF literal contract supports them.
- Known property domain and range declarations are respected where compatibility can be determined without reasoning.
- A class cannot be made a superclass of itself.
- Duplicate superclass relationships are rejected.
- Existing direct superclass relationships are required for removal.
- The proposal baseline is current.
- Temporary Turtle serialization and reparsing are semantically equivalent.

Validation must produce existing structured `ValidationIssue` values with stable codes, severity, message, and source fields. The engine must remain deterministic and must not perform full OWL reasoning.

## Error Behavior

- Invalid CLI input returns a nonzero exit code and a machine-readable error or validation payload.
- Invalid workbench form input is shown near the relevant field where practical and is also rejected by the Kotlin boundary.
- Unknown edit kinds return a structured unsupported-edit issue.
- Missing target sources or referenced entities return deterministic validation issues.
- Duplicate additions and missing removals prevent approval and application.
- Incompatible property values prevent approval and application.
- Stale proposals return a stale/apply-failed result without overwriting the changed source.
- Rejection returns a rejected result and leaves source files unchanged.
- Serialization, reparsing, or post-save verification failures return structured failure and rollback results.
- A rollback failure is reported separately from the original apply failure.

No error path may write ontology files before the proposal has passed validation, semantic-equivalence verification, and approval.

## Test Cases

Each edit kind should have focused tests for:

- Typed-edit translation into the expected graph changes.
- CLI argument parsing and unsupported or incomplete input.
- Machine-readable preview, validation, diff, apply, and reject payloads.
- Preview graph behavior without source mutation.
- Semantic diff entries for additions, removals, and label replacement.
- Validation success and deterministic validation failures.
- Rejection without file changes.
- Successful application, reload, semantic equivalence, and affected-file reporting.
- Stale baseline blocking where relevant.
- Serialization and post-save rollback behavior.
- VS Code request models, form rendering, process invocation, result handling, and refresh behavior.

End-to-end tests should use temporary copies of `examples/simple-ontology` or small temporary fixtures. They must not modify committed example files in place.

Required end-to-end coverage should include at least:

- Object property with domain and range.
- Datatype property with a typed value.
- Individual creation and type assignment.
- Object-property assertion.
- Datatype-property assertion.
- Superclass addition and removal.
- Label addition or replacement.
- Reject, stale, and rollback paths.

## Acceptance Criteria

Phase 2.5 is accepted when:

- Every listed Phase 2.5 edit kind can be requested through the machine-readable CLI.
- Every listed Phase 2.5 edit kind has a focused VS Code form or form mode.
- The existing `create-class` flow remains backward compatible.
- Every edit uses the existing proposal preview, diff, validation, approval, atomic save, reload, and rollback pathway.
- Users can see a semantic diff and validation result before approval.
- Rejection leaves the project unchanged.
- Stale proposals are blocked before writes.
- Successful application changes only the intended ontology source and survives reload with semantic equivalence.
- Save or verification failures restore the prior source and expose rollback status.
- CLI, Kotlin module, and VS Code tests cover successful and failure paths.
- End-to-end tests use copied or temporary fixtures and leave committed examples unchanged.
- `./gradlew test`, `./gradlew build`, and `./gradlew check` pass.
- The extension compile and test commands pass in a Node-enabled environment.
- No new RDF, OWL, SHACL, Turtle, reasoning, or Git framework is introduced.

## Open Questions

- Should the CLI continue with edit-specific picocli options, or should a later slice introduce one structured JSON request while preserving the current commands?
- Should property domain and range forms support one value initially or multiple asserted values in one proposal?
- Should a create-individual form support multiple type assignments in one change set or require repeated proposals?
- Which literal lexical forms should be accepted for each supported datatype, and should invalid values be rejected before the CLI boundary?
- Should label replacement always be represented as one composed removal and addition, or should a dedicated typed edit be added?
- How should the workbench select entities across multiple ontology sources while preserving one target source for writes?
- Should deterministic superclass cycle detection be required for Phase 2.5 or deferred until a later reasoning-focused phase?
- Should the workbench display inferred compatibility information, or only explicitly asserted domain and range statements?
- How should blank-node resources be represented in assertion forms without implying durable identity?

## Boundary Check

Phase 2.5 fits the current project direction because it completes the user-facing editing surface already intended by Phase 2 and reuses the existing Kotlin/JVM semantic engine.

It does not add web, database, AI, document, Git, full reasoning, or full OWL/SHACL infrastructure. It keeps RDF and ontology interpretation in established libraries and keeps the VS Code extension as a thin client of the Kotlin engine.
