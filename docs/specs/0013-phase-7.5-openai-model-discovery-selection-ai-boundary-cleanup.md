# Feature Spec: Phase 7.5 OpenAI Model Discovery, Selection, And AI Boundary Cleanup

## Status

Draft for review. Phase 7 is implemented through the current tool-driven AI boundary, and Phase 7.5 is an additive planning refinement. This document specifies Phase 7.5 behavior but does not authorize implementation.

## Problem

Entio currently requires every user of the native AI assistant to use one server-approved OpenAI model, `gpt-5.2`. A credential can be valid while the API project that issued it cannot access that model. The current fixed-model assumption therefore prevents otherwise valid users from using Entio and conflates credential validity with model availability.

Entio must distinguish invalid credentials from valid credentials that expose zero, one, or several Entio-compatible models. It must also handle a previously verified model becoming inaccessible without deleting a still-valid credential, losing private work, silently selecting another model, or affecting non-AI features.

The Phase 6 and Phase 7 AI implementation also grew across provider, credential, capability, conversation, draft, audit, and web-contract concerns in a mostly flat tree. Adding discovery and selection without first documenting and, where justified, separating those responsibilities would make credential and provider state harder to reason about and test.

## Central Product Promise

> A user can connect their own OpenAI API key, see the models that are both available to their API project and approved for Entio's tool-driven ontology workflow, explicitly select and verify one, and safely use it without changing ontology configuration or exposing credentials or raw provider inventory to the browser.

## Goals

- Replace the single pinned-model product requirement with server-side, per-user OpenAI model discovery and explicit selection.
- Discover models using only the current user's server-memory credential.
- Intersect provider-visible models with an Entio-owned catalog and compatibility policy in Kotlin.
- Return only curated Entio model descriptors to React.
- Require explicit selection and bounded verification before marking AI ready, including when only one eligible model exists.
- Store the verified selection per user in server memory, independently of projects and ontology files.
- Allow refresh, retest, and reselection without requiring re-entry of an unchanged valid credential.
- Bind every AI run to one verified model for the lifetime of that run.
- Detect model removal or access loss and require explicit reselection while preserving conversations and private drafts.
- Preserve the provider-neutral boundary and keep OpenAI mapping in the OpenAI adapter.
- Document and, only where useful, reorganize the AI subsystem into clear ownership boundaries.
- Preserve all Phase 1 through Phase 7 semantic, proposal, review, collaboration, CLI, VS Code, and web behavior.

## Non-Goals

Phase 7.5 does not include:

- Free-text or arbitrary model identifiers.
- Automatic approval of every model returned by OpenAI.
- Automatic or silent model selection, fallback, or mid-run switching.
- Browser-to-OpenAI requests or raw provider responses in browser contracts.
- Durable credentials, selections, conversations, drafts, or audit storage.
- Model selection in `entio.yaml`, project registration, ontology sources, or collaboration state.
- Shared user selections, organization administration, billing controls, or exact live pricing.
- Model benchmarking, fine-tuning, embeddings, vector stores, or task-based automatic routing.
- A multi-provider product UI; the internal interface remains provider-neutral.
- Expert controls for arbitrary reasoning, verbosity, timeout, endpoint, or provider parameters.
- Changes to the AI capability registry's semantic operations.
- Changes to typed ontology or SHACL edits, private draft semantics, deterministic analysis, proposal review, application, or rollback.
- Changes to CLI or VS Code behavior.
- A new Gradle module or broad unrelated server/client refactoring.

Current phase-level architecture and historical non-goals remain governed by `AGENTS.md`.

## Existing Foundation

Phase 7.5 builds on:

- The `web-server` Ktor boundary, development identity, per-user in-memory credential store, provider-neutral AI interfaces, OpenAI Responses adapter, conversations, runs, capabilities, drafts, audits, and review submission.
- The `web-app` credential settings, assistant panel, conversation and draft-review surfaces, versioned transport clients, and query state.
- Existing Kotlin semantic services, which remain authoritative and are not model-selection implementation targets.
- Existing deterministic provider doubles and Ktor mock transport tests, which remain the CI boundary for provider behavior.

## Fixed Architecture Decisions

### Kotlin owns eligibility and selection

React may request discovery, render returned choices, and submit a selected ID. Kotlin decides whether that ID is currently visible, catalog-approved, compatible, enabled, non-deprecated, and eligible. The client cannot expand the catalog or compatibility policy.

### Discovery and compatibility are separate

The selectable set is:

```text
models visible to the current credential
intersection Entio's enabled model catalog
intersection the current Entio Responses/tool compatibility policy
= selectable models
```

A provider-visible model does not become selectable merely because it appears in `GET /v1/models`.

### Explicit selection is always required

Entio does not auto-select a model, even when discovery returns exactly one eligible model. The user must select it and acknowledge that verification may incur a small API charge.

### Provider endpoint and catalog are server controlled

The production OpenAI endpoint remains fixed and server-owned. The Entio model catalog is application configuration in `web-server`; it is not ontology configuration, browser configuration, or AI-editable state.

### Selection is session state

Credential secrets remain in the existing secret store. Model discovery and selection state is separate, per user, in memory, cleared on restart, and never shared with collaborators. Replacing or removing the credential clears the selection.

### One model per run

The selected model is copied into the AI run at creation. The run uses that model throughout tool calls, self-correction, draft analysis, and streaming. A later user selection affects only future runs.

### Cleanup is bounded

The AI tree may be reorganized to separate provider, credentials, models, capabilities, conversations, drafts, context, audit, and web mapping. Cleanup must preserve behavior, avoid semantic modules, retain deterministic test doubles, and stop short of cosmetic churn or a new module.

## Proposed Behavior

### 1. Credential submission and discovery

When a user submits or replaces an OpenAI API key:

1. React sends the key once to Ktor and clears the input after the request completes.
2. Ktor normalizes and stores the key in the current user's server-memory credential store.
3. Any prior selected model and verification are invalidated.
4. Ktor calls OpenAI `GET /v1/models` through the provider discovery interface.
5. The OpenAI adapter maps the response immediately to minimal provider-neutral descriptors.
6. The discovery service intersects visible IDs with the Entio model catalog and compatibility policy.
7. Ktor returns credential, discovery, and selection status plus only selectable Entio descriptors and a bounded unsupported-model count.
8. AI remains unavailable until the user selects and verifies a model.

Saving a syntactically non-empty credential does not by itself produce a ready state.

### 2. Entio model catalog

The server owns a deterministic model catalog. Each entry contains:

- Exact stable OpenAI model ID; family rules require a separately reviewed catalog change.
- Entio display name and description.
- Provider ID.
- Enabled and deprecation status.
- Responses API, custom function-tool, streaming, and required structured-output compatibility.
- Entio capability tier.
- Server-owned request defaults such as reasoning effort, response verbosity, context policy, token policy, and timeout class where supported.
- Relative speed and cost descriptions without exact price claims.
- Preference rank and recommended status.
- Optional replacement recommendation for deprecated entries.

Catalog ordering is deterministic: preference rank, display name, then model ID. Unknown or disabled provider models never become selectable automatically.

### 3. Eligibility states

Internally, catalog/provider intersection supports:

- `AVAILABLE_AND_COMPATIBLE`
- `AVAILABLE_BUT_DISABLED`
- `AVAILABLE_BUT_INCOMPATIBLE`
- `APPROVED_BUT_NOT_AVAILABLE`
- `DEPRECATED`
- `UNKNOWN_PROVIDER_MODEL`

Only `AVAILABLE_AND_COMPATIBLE` entries appear as selectable model descriptors. The browser may receive the count of omitted provider models but not their unrestricted identities or raw metadata.

### 4. Model selection and verification

The user selects an ID returned in the current selectable set. Before verification, the server rechecks discovery freshness and eligibility. If discovery is older than 15 minutes, selection refreshes it first.

Verification sends a minimal, non-project Responses request containing an Entio-owned harmless custom function schema. It proves that the selected model accepts the request shape, function-tool contract, streaming/response mapping, and required provider configuration. It does not inspect a project, include ontology content, mutate a draft, or create a proposal.

The selection becomes `READY` only after verification succeeds. Verification failure preserves the credential when authentication remains valid and reports a structured reason. Repeated requests with the same idempotency key do not duplicate verification.

### 5. Existing credential operations

A user with a stored credential can:

- Inspect status without receiving the secret.
- Refresh models.
- Select and verify an eligible model.
- Retest the selected model.
- Change to another eligible model.
- Clear the selection while preserving the credential.
- Replace or remove the credential.

Replacing or removing a credential invalidates discovery and selection state. Removing it also returns the assistant to `NOT_CONFIGURED`.

### 6. Discovery freshness

Discovery runs:

- Immediately after credential submission or replacement.
- On explicit Refresh models.
- Before selection when the cached result is older than 15 minutes.
- Before the first AI run when no fresh discovery exists after server restart.
- After model-not-found or model-access-denied provider failures.

Normal tool-loop calls do not call `GET /v1/models`. The initial freshness interval is 15 minutes and remains server-owned configuration.

### 7. AI readiness and run binding

AI readiness requires all of:

- A configured credential that is not known invalid.
- Completed discovery with at least one eligible model.
- A selected model still present in the eligible set.
- Successful verification for that credential/model combination.
- Verification not invalidated by credential replacement, removal, or availability refresh.

At run creation, Kotlin copies the selected model ID and its catalog request policy into immutable run/provider request state. Audit records may include the exact model ID and catalog/prompt version but never credentials.

### 8. Model becomes unavailable

Before a new run, stale or missing discovery is refreshed. If the selected model is absent, disabled, deprecated, or incompatible, the run does not start. Entio marks the selection `UNAVAILABLE` or `INCOMPATIBLE`, preserves the credential, conversations, and private drafts, and requires reselection.

If an active run receives model-not-found or access-denied:

- The run stops safely with a structured model-unavailable result.
- No retry uses another model.
- Existing draft and conversation state remain intact.
- The selection is invalidated and discovery is refreshed.
- The UI offers model reselection for a future run.

### 9. No compatible models

A valid credential with no selectable intersection produces `NO_COMPATIBLE_MODELS`, not `INVALID`. The UI explains that the API project does not currently expose a model supported by Entio and offers Refresh models, Replace key, and Remove key. Non-AI workbench features remain usable.

### 10. React experience

The settings surface supports:

- No credential.
- Saving credential.
- Discovering models.
- Valid credential with selection required.
- Valid credential with no compatible models.
- Selecting and verifying.
- Ready.
- Stale or unavailable selection.
- Invalid credential.
- Rate limited, timeout, discovery failure, and provider unavailable.
- Credential removed.

Each selectable model displays Entio-owned name, exact ID as secondary text, recommended badge when applicable, capability tier, relative speed/cost language, and a short Entio-focused description. The UI provides Select and test, Change model, Refresh models, Replace key, and Remove key actions. It discloses that verification may incur a small API charge.

The assistant panel shows ready only when selection status is `READY`; configured credentials without a verified selection are not labeled ready.

### 11. AI subsystem cleanup and map

Implementation begins with an inventory of the existing AI packages, entry points, state stores, and dependency directions. If movement materially improves ownership, server code may be grouped under provider/openai, credentials, models, capabilities, conversation, draft, context, audit, and web packages, and frontend AI code may be grouped by credentials, models, conversations, drafts, and shared UI.

The cleanup produces `docs/architecture/ai-subsystem-map.md`, identifying responsibilities, state ownership, boundaries, test doubles, and forbidden dependency directions. Every structural move must preserve public behavior and pass focused tests before model-selection behavior is added.

## Inputs

### User inputs

- OpenAI API key, submitted once over the existing authenticated/development-user Entio boundary.
- Explicit Refresh models action.
- Model selection from the server-returned eligible set.
- Explicit verification/retest action and idempotency key.
- Replace-key, remove-key, or clear-selection action.

### Server configuration

- Fixed OpenAI API host and model-list/Responses paths.
- Entio model catalog and catalog version.
- Compatibility policy version.
- Prompt version and model-specific request defaults.
- 15-minute discovery freshness interval.
- Existing provider timeouts, safe retry limits, and per-user abuse limits.

The browser cannot supply provider endpoints, catalog entries, compatibility flags, or expert tuning values.

## Outputs

Versioned Entio responses expose only server-owned state such as:

- Credential status.
- Discovery status and last checked time.
- Ordered selectable model descriptors.
- Unsupported provider-model count.
- Selected model descriptor.
- Selection and verification status.
- User-facing structured error and available actions.

AI run and audit outputs include the run-bound model ID where appropriate. Outputs never include the credential, authorization header, raw OpenAI model object, raw provider body, hidden compatibility rules, or another user's settings.

## Web Contract Behavior

Phase 7.5 extends the existing `/api/v1/ai` credential boundary rather than introducing duplicate user-resource routes. The target contract is:

```text
GET    /api/v1/ai/provider-settings
PUT    /api/v1/ai/credentials
DELETE /api/v1/ai/credentials
POST   /api/v1/ai/models/discover
GET    /api/v1/ai/models
PUT    /api/v1/ai/model-selection
POST   /api/v1/ai/model-selection/test
DELETE /api/v1/ai/model-selection
```

Existing credential-status and credential-test routes remain backward compatible during migration and may delegate to the unified settings service. New mutation endpoints require the current user and idempotency keys where provider calls could otherwise repeat.

## State Model

The server owns an `AiUserProviderSettings`-equivalent record containing no secret:

```text
userId
providerId
credentialStatus
discoveryStatus
discoveredAt
catalogVersion
selectableModelIds
unsupportedProviderModelCount
selectedModelId
selectedModelVerifiedAt
selectionStatus
lastProviderErrorCategory
```

The API key remains only in the credential store. Settings are removed on logout/session cleanup or restart and are isolated by user.

## Validation Behavior

Validation is deterministic for the same catalog, provider descriptor set, and request:

- Provider ID and endpoint must match server configuration.
- Credential input must be non-empty after normalization.
- Provider model IDs are mapped as data and never executed as configuration.
- Catalog entries must have unique exact IDs, deterministic ranks, supported provider IDs, and internally consistent compatibility flags.
- Returned model choices are the server-computed eligible subset.
- Selected IDs must exist in the current user's fresh eligible set.
- Verification must use the selected catalog entry and server-owned request policy.
- An AI run requires a verified selection for the same credential generation.
- Cross-user settings, selection, discovery, and credential access are rejected.
- Replayed idempotency keys return the prior result without repeating billable verification.
- Discovery and verification are rate limited per user.

## Error Behavior

Structured error codes include:

- `AI_CREDENTIAL_MISSING`
- `AI_CREDENTIAL_INVALID`
- `AI_MODEL_DISCOVERY_FAILED`
- `AI_NO_COMPATIBLE_MODELS`
- `AI_MODEL_SELECTION_REQUIRED`
- `AI_MODEL_NOT_AVAILABLE`
- `AI_MODEL_NOT_APPROVED`
- `AI_MODEL_INCOMPATIBLE`
- `AI_MODEL_VERIFICATION_FAILED`
- `AI_MODEL_BECAME_UNAVAILABLE`
- `AI_PROVIDER_RATE_LIMITED`
- `AI_PROVIDER_UNAVAILABLE`
- `AI_PROVIDER_TIMEOUT`

Provider authentication failure marks the credential invalid and clears selection. Model-specific access failure preserves a valid credential and invalidates only the selection. Timeout, rate limit, malformed provider response, and outage are redacted and do not expose provider bodies. Every AI failure leaves non-AI routes healthy.

## Security And Privacy Requirements

- All provider calls originate in `web-server`.
- Credential secrets remain callback-scoped inside the server credential boundary.
- Model discovery and verification use only the current user's credential.
- Provider DTOs are mapped before crossing into Entio services or web DTOs.
- Raw provider payloads, authorization headers, and credentials are never logged or returned.
- Unknown provider fields are ignored or rejected safely.
- The client cannot submit an arbitrary provider, endpoint, or non-eligible model.
- The AI capability loop cannot call discovery, edit the catalog, or change selection.
- Cross-user access is rejected and covered by tests.
- Browser storage contains status and descriptors only; credential inputs are cleared after submission.
- Discovery and verification limits prevent repeated provider-call abuse.

## Test Cases

### Catalog and compatibility

- Catalog construction rejects duplicate IDs and inconsistent entries.
- Identical provider inventory produces byte-for-byte stable eligible ordering.
- Unknown, disabled, incompatible, deprecated, and unavailable models are not selectable.
- Unsupported provider inventory is represented only by a count.
- React cannot add or alter compatibility metadata.

### Discovery

- A valid credential maps provider inventory into the approved subset.
- Discovery uses Bob's key without affecting Alice's state.
- Invalid credential, rate limit, timeout, malformed JSON, and outage map to structured states.
- Raw provider fields and error text never reach web responses.
- Fresh results are reused within 15 minutes; explicit refresh bypasses freshness.

### Selection and verification

- Zero eligible models produces `NO_COMPATIBLE_MODELS` while preserving credential validity.
- One eligible model still requires explicit selection.
- Multiple eligible models are ordered and selectable.
- Non-eligible and arbitrary IDs are rejected before a provider call.
- Verification uses a harmless tool request with no project or ontology content.
- Verification success stores the selection; failure preserves a valid credential.
- Repeated idempotent verification does not repeat the provider call.
- Replacing/removing a key or restarting clears selection.
- One user's selection never affects another user.

### Runtime

- Runs cannot start with no selection or an unverified selection.
- Each run captures one model ID and keeps it across all provider requests.
- Changing selection affects only later runs.
- Model-not-found/access-denied stops safely, preserves drafts, invalidates selection, and triggers refresh.
- No automatic fallback or mid-run switching occurs.

### UI

- Credential submission clears the key input and enters discovery state.
- Selection-required, no-compatible-models, verifying, ready, unavailable, invalid, rate-limited, timeout, and provider-down states render accessibly.
- Only eligible descriptors appear in the selector.
- Verification charge disclosure is visible.
- Refresh, select/test, change, retest, replace, remove, and clear actions work by keyboard and screen reader.
- Assistant readiness reflects verified selection, not credential presence alone.

### Cleanup and regression

- Structural moves preserve existing HTTP and AI behavior.
- Deterministic fake providers remain available.
- No semantic module gains provider or web dependencies.
- All Phase 1 through Phase 7 tests remain green.
- Primary journeys for multiple models, no compatible models, change model, unavailable model, and credential replacement pass end to end.

## Acceptance Criteria

Phase 7.5 is accepted when:

1. Credential submission triggers server-side model discovery through the current user's key.
2. React receives only deterministic Entio-approved model descriptors and a bounded unsupported count.
3. Unknown, disabled, incompatible, deprecated, and unavailable models cannot be selected.
4. The user explicitly selects and verifies a compatible model before AI becomes ready.
5. Verification is minimal, non-project, safely disclosed, and cannot mutate Entio state.
6. Selection is per-user server-memory state and is cleared by credential replacement/removal and restart.
7. Every AI run captures one verified model and never switches during its lifetime.
8. Model access loss stops safely, preserves conversation/draft state, and requires explicit reselection.
9. Credential secrets, authorization headers, raw provider payloads, and unrestricted model inventory never reach React, logs, events, drafts, proposals, snapshots, or audits.
10. Provider failures do not affect non-AI workbench behavior.
11. `docs/architecture/ai-subsystem-map.md` accurately documents AI ownership and dependency boundaries.
12. Cleanup preserves public behavior and does not modify semantic or proposal policy.
13. Focused provider, catalog, discovery, selection, verification, runtime, security, UI, and end-to-end tests pass.
14. All existing Phase 1 through Phase 7 regressions pass.

## Open Questions

- Which exact OpenAI model IDs form the initial catalog? This must be approved immediately before the catalog slice using current provider documentation and contract tests; the spec intentionally does not bless moving aliases or arbitrary families.
- Which catalog entries should be marked recommended? Recommendation is an Entio-owned product decision and must not be inferred solely from provider ordering.
- Should the 15-minute discovery freshness interval become deployment-configurable after Phase 7.5? Phase 7.5 keeps it server-owned and fixed unless implementation evidence requires a bounded configuration field.
- What per-user discovery and verification rate limits fit local development and future production identity? The ExecPlan requires explicit bounded defaults and tests before exposing endpoints.
- Can existing credential routes be extended without confusing Phase 6/7 clients, or is a short compatibility window with both status shapes required? The contract slice must resolve this from current web-client usage before removal of any route.

## Boundary Check

- Phase fit: additive Phase 7 provider/credential refinement; no later-phase infrastructure is required.
- Non-goals: no durable persistence, billing administration, arbitrary model routing, multi-provider UI, or semantic changes.
- Dependency direction: provider, session, and HTTP behavior remains in `web-server`; React remains presentation-only; semantic modules remain independent.
- Standards: no RDF, OWL, or SHACL implementation changes are introduced.
- Speculative infrastructure: no new module, database, queue, provider SDK, pricing service, or broad framework is required.
