# ExecPlan: Phase 2.5+ Label-First Editing, Deletion, And Staged Change Sets

## Status

Draft

## Source Spec

- `docs/specs/0005-phase-2.5-plus-ontology-workbench-usability.md`
- `docs/architecture/phase-2.5-plus-scope.md`

## Objective

Extend the completed Phase 2.5 editing workflow so users can identify ontology elements by label, receive deterministic generated IRIs, review supported deletions and their dependencies, stage multiple edits, and apply one combined approved change set atomically.

The implementation must preserve the Kotlin semantic engine as the source of truth. The VS Code extension remains a thin client for session state and rendering, and the CLI remains a thin machine-readable boundary.

## Current State

- `core-types` defines RDF terms, graph triples and changes, typed ontology edits, proposal contracts, validation reports, diffs, apply results, and symbol details.
- `semantic-engine` loads projects, parses Turtle through Apache Jena, extracts symbols and relationships, translates typed edits, creates previews, verifies Turtle round trips, and applies approved proposals with rollback.
- `validation-engine` validates project and proposal state using deterministic explicit graph checks.
- `graph-diff` produces semantic graph and proposal diffs.
- `cli` exposes single-edit proposal preview, validation, diff, apply, and reject commands through picocli and structured output.
- `vscode-extension` provides label/detail rendering, typed edit forms, proposal preview/action flow, refresh, and changed-source opening, but does not yet provide the Phase 2.5+ label-first creation workflow, deletion UI, or multi-edit staging.
- `examples/simple-ontology` provides a small committed fixture; tests must copy it before mutation.

## Target State

- Project configuration can supply a validated `iriNamespace` for generated local identifiers.
- Kotlin resolves labels, generates and collision-checks IRIs, analyzes deletion dependencies, normalizes staged changes, detects conflicts, and prepares one combined proposal.
- The CLI accepts a structured ordered multi-edit request while preserving existing single-edit commands.
- The VS Code workbench uses label-backed controls, displays generated IRIs, supports safe deletion review, and maintains an in-memory staged list with edit/remove/re-preview behavior.
- Combined preview, validation, semantic diff, Turtle equivalence, approval, atomic apply, reload, refresh, rejection, stale detection, and rollback work as one lifecycle.
- Tests cover deterministic contracts, CLI and workbench boundaries, copied fixtures, and failure paths.

## Scope

### Affected Modules And Files

Expected implementation areas are listed below. Exact filenames may be refined within a slice if existing ownership makes a nearby file more appropriate:

- `core-types/src/main/kotlin/com/entio/core/`
  - Namespace, label-resolution, generated-IRI, deletion-dependency, staged-change, conflict, and combined-proposal contracts.
- `semantic-engine/src/main/kotlin/com/entio/semantic/`
  - Project namespace access, deterministic IRI generation, label resolution, deletion analysis/change creation, staged normalization, combined preview preparation, and existing proposal services.
- `validation-engine/src/main/kotlin/com/entio/validation/`
  - Generated-IRI, deletion, staged-conflict, attribution, and combined-proposal checks.
- `graph-diff/src/main/kotlin/com/entio/diff/`
  - Only focused support needed to present one combined semantic diff.
- `cli/src/main/kotlin/com/entio/cli/`
  - Structured request parsing, label/IRI/dependency operations, combined proposal dispatch, and JSON output.
- `vscode-extension/src/`
  - Label-first selectors/details, delete/dependency UI, staged session model, form restore/edit/remove behavior, combined preview/actions, and refresh.
- `examples/` and test temporary directories
  - Copied fixtures only; committed examples must not be mutated by tests.
- `docs/decisions/`
  - Slice completion artifacts or focused ADRs only when an implementation decision is architecturally significant.

### Recommended Dependency Strategy

- Reuse existing Gradle module dependencies and Apache Jena for graph semantics.
- Add no new semantic-web, OWL, SHACL, server, database, or UI framework dependency.
- If the existing CLI has no suitable JSON parser, add one narrowly to `cli`, preferably `com.fasterxml.jackson.module:jackson-module-kotlin` at a pinned version selected during implementation. Keep JSON parsing out of semantic-engine and core-types.
- Use the existing TypeScript toolchain. Do not add a UI framework or state-management dependency.
- Do not add persistence dependencies for staged sessions.

## Non-Goals

- AI, document ingestion, Schema RAG, entity resolution, autonomous agents, or production graph storage.
- Full OWL reasoning, full SHACL, full Protégé parity, or unsupported OWL constructs.
- Source-text-preserving Turtle editing.
- Long-term staged-change persistence, multi-user review, authentication, authorization, servers, or databases.
- Actual Git operations inside Entio.
- New Gradle modules or a second source mutation path.

## Dependency Order And Multi-Agent Safety

Implement slices serially in the order below. The early contracts and semantic services are shared foundations; later CLI and workbench slices must consume them rather than redefine them.

1. Core namespace, label-resolution, deletion, staging, and combined-proposal contracts.
2. Deterministic namespace validation, label resolution, and IRI generation.
3. Deletion dependency analysis and explicit deletion change generation.
4. Staged change normalization, conflict detection, and combined preview preparation.
5. Structured CLI request/response boundary and label/IRI/dependency operations.
6. CLI combined proposal preview, validation, diff, apply, reject, stale, and rollback flows.
7. VS Code label-first selectors, generated-IRI details, and delete/dependency display.
8. VS Code staged-change session, edit/remove/cancel, form clearing, and combined preview state.
9. VS Code approval/rejection/apply/refresh integration for combined change sets.
10. Deletion proposal selection and CLI boundary completion.
11. VS Code deletion selection and proposal flow.
12. End-to-end Phase 2.5+ regression, fixture-copy tests, and documentation summary.

Slices 1 through 4 must be serial because they establish shared contracts and semantic behavior. Slices 5 and 6 must be serial because they share CLI request and output boundaries. Slices 7 through 11 must be serial because they share the workbench message and form model. Slice 12 must run last.

After Slice 4 establishes stable contracts, narrowly scoped test-only work may be parallelized when each agent owns separate test files. Do not parallelize edits to `core-types`, `shared`, build files, CLI JSON contracts, workbench message models, specs, ExecPlans, or `AGENTS.md`.

## Rollback Notes

- Each implementation slice must remain independently revertible.
- If a slice fails verification, leave its branch unmerged and restore the working tree to the previous local base before asking how to proceed.
- Do not reset or discard unrelated user changes.
- For code rollback after merge, revert the slice commit(s) rather than editing later slices to hide the behavior.
- Runtime apply failures must continue to use `ProposalApplier`'s existing source backup and restoration path.
- Staged session state is in memory only, so closing the workbench discards it without source changes.

## Risks And Assumptions

- The existing project configuration has no namespace field, so adding `iriNamespace` must preserve loading of older projects and make the missing value explicit for generated-ID operations.
- Labels may be duplicated across kinds or ontology sources; deterministic candidate ordering and explicit disambiguation are required.
- Deletion dependencies are limited to explicit graph statements and do not represent inferred OWL consequences.
- Combined staged edits may need richer attribution than current proposal responses; extend contracts narrowly rather than exposing library types.
- A JSON request parser may be needed in `cli`; it must remain isolated to that module and be justified as a boundary dependency.
- The VS Code extension currently has session state but no durable proposal store; this plan intentionally preserves that limitation.
- Multiple ontology sources create target-source ambiguity; initial combined application should require one target source unless a later approved design supports source-partitioned atomic application.

## Definition Of Done

- All twelve slices are implemented in dependency order and each has focused tests, verification, a completion artifact, and an independently reviewable commit.
- Label-first resolution, deterministic IRI generation, collision handling, supported deletion, workbench deletion selection, dependency review, multi-edit staging, staged edit/remove, combined preview, and atomic apply are implemented as specified.
- The Kotlin engine remains the sole semantic and persistence authority.
- CLI and VS Code boundaries consume the same structured behavior without duplicating RDF logic.
- Rejection, stale, validation-failure, round-trip-failure, and rollback paths leave sources unchanged.
- Copied-fixture end-to-end tests pass without modifying committed examples.
- `./gradlew test`, `./gradlew build`, `./gradlew check`, and `vscode-extension` `npm test` pass.
- The Phase 2.5+ summary accurately describes the actual repository state and calls out deviations or deferred open questions.
- No web, AI, document, Git, database, full reasoning, full OWL/SHACL, or speculative infrastructure was introduced.

## Boundary Check

This plan fits the requested Phase 2.5+ scope and the repository architecture. It keeps shared contracts in `core-types`, semantic behavior in `semantic-engine` and `validation-engine`, diff behavior in `graph-diff`, orchestration in `cli`, and session/rendering behavior in the VS Code extension. It reuses Apache Jena and existing proposal services rather than creating custom RDF/OWL/SHACL infrastructure, and it does not add Entio-managed Git workflow or long-term staged-change persistence.
