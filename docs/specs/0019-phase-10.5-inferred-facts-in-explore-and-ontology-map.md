# Feature Spec: Phase 10.5 Inferred Facts In Explore And Ontology Map

## Status

Draft for approval.

## Problem

Entio already computes bounded OWL reasoning results and distinguishes asserted facts from inferred facts. Today, users primarily see those results in the Reasoning workspace. Explore and the read-only ontology map continue to show asserted knowledge only.

This separation makes ordinary ontology browsing incomplete: a user must leave the class, property, individual, or map they are inspecting to understand what the current applied ontology entails.

Phase 10.5 adds a read-only inferred-facts overlay to Explore:

```text
complete current applied-graph reasoning result
→ Kotlin extracts supported inferred facts
→ server verifies scope, freshness, completeness, and limits
→ Explore merges provenance-aware read entries
→ ontology map layers inferred edges over asserted layout
```

The ontology source remains unchanged. Viewing an inference does not stage, materialize, approve, or apply it.

## Goals

- Add one `Inferred facts` toggle to the existing Project Outline filter popover.
- Keep the toggle off by default.
- Apply the same toggle to the Project Outline, entity details, and ontology map within the current project.
- Show supported current inferred facts in the same fields as equivalent asserted facts.
- Mark every inferred entry with a light-blue treatment and explicit `Inferred` text or accessible metadata.
- Show inferred superclass/subclass relationships, individual types, and object-property assertions.
- Suppress inferred duplicates when the same canonical fact is asserted.
- Add clearly distinguished inferred edges to the bounded ontology map.
- Keep asserted class hierarchy responsible for primary map placement.
- Detect unavailable, loading, failed, incomplete, and stale reasoning states.
- Keep Kotlin authoritative for semantic extraction, identity, provenance, ordering, deduplication, freshness, authorization, and limits.
- Preserve current Explore, Reasoning, map, staging, proposal, CLI, and VS Code behavior when the toggle is off.

## Non-Goals

- Staging, applying, approving, or materializing inferred facts from Explore or the map.
- Automatic or background reasoning when the toggle is enabled.
- New inference rules, a new reasoner, or browser-side inference.
- Inferred domains, ranges, property characteristics, datatype values, annotations, equivalence axioms, same-as facts, negative assertions, anonymous expressions, or SHACL findings.
- Inferred literal nodes in the map.
- Editing or deleting inferred facts.
- Ontology source writes or changes to the proposal workflow.
- Durable reasoning-result or filter-state persistence.
- CLI or VS Code inferred-fact visualization.
- New modules, dependencies, databases, queues, global state frameworks, or semantic grouping.
- Unbounded reasoning-result delivery to the browser.

Current repository boundaries and phase-level non-goals remain governed by `AGENTS.md`.

## Proposed Behavior

### 1. Eligible Reasoning Result

The overlay may use only one server-retained reasoning result that:

- belongs to the current project;
- was produced for the applied graph, not a proposal preview;
- has completed successfully;
- is marked complete;
- includes a complete import closure under the existing reasoning contract;
- matches the current applied-graph fingerprint;
- is visible to the current user under existing project and source permissions.

The server selects the newest eligible result deterministically by completion time and job ID as the final tie-breaker. A result from another project, user-inaccessible scope, proposal graph, failed job, cancelled job, incomplete job, or older graph fingerprint is never exposed as current inferred knowledge.

The retained result remains the source of the overlay. Phase 10.5 does not rerun reasoning for each Explore request.

### 2. Filter Placement And Lifetime

The existing Project Outline filter popover gains a checkbox labeled `Inferred facts`.

The state is:

- off when a project is first opened;
- shared by the Project Outline, entity-detail views, search-visible outline content, and ontology map for that project;
- preserved while switching Explore tabs in the same open project;
- reset to off on page reload, session restart, or project change;
- not stored in ontology, server project state, browser local storage, or user settings.

Closing the filter popover does not change the selection. `Reset filters` restores existing filter defaults and turns `Inferred facts` off.

The control has these presentation states:

| State | Behavior |
| --- | --- |
| `Off` | Asserted-only Explore behavior; inferred requests are not made where avoidable. |
| `Loading` | Toggle remains on, asserted content stays usable, and a bounded loading message is shown. |
| `On` | Current supported inferred facts are merged into Explore. |
| `Unavailable` | No current applied-graph reasoning result exists; inferred facts remain hidden. |
| `Stale` | A retained result exists but its fingerprint is not current; inferred facts remain hidden. |
| `Failed` | The relevant reasoning job failed, was cancelled, or cannot be read; inferred facts remain hidden. |

Unavailable and stale messages include a link or button that opens the existing Reasoning workspace. They do not start reasoning automatically.

### 3. Supported Facts And Canonical Identity

Phase 10.5 supports the same three named-resource fact families as Phase 10 materialization:

| Fact family | Canonical relationship | Explore use |
| --- | --- | --- |
| Class relationship | `subject rdfs:subClassOf superclass` | superclass, subclass, and hierarchy fields |
| Individual type | `individual rdf:type class` | individual type and class membership fields |
| Object-property assertion | `subject predicate object` | property usage and incoming/outgoing individual relationships |

All subject, predicate, and object terms must be named IRIs. Unsupported or malformed reasoning outputs remain available only through the existing Reasoning workspace.

Kotlin derives stable fact identity from the existing Phase 10 canonical semantic fact key where available. Otherwise it uses the same canonical tuple and encoding:

- supported fact family;
- canonical subject IRI;
- canonical predicate IRI;
- canonical object IRI.

Labels, rendering location, source display names, and ordering do not affect identity.

### 4. Provenance-Aware Read Model

Existing Explore and graph read contracts are extended additively. A displayed fact carries:

- stable semantic fact identity;
- subject IRI;
- predicate IRI or relationship kind;
- object/target IRI;
- `Asserted` or `Inferred` origin;
- source identity where meaningful;
- reasoning job/result reference for inferred facts;
- applied-graph fingerprint;
- freshness state;
- supported inference type.

Asserted and inferred collections remain separable in Kotlin and server contracts. React may merge them for presentation only according to server-provided identity and order; it must not derive relationships or decide that two different triples are equivalent.

Existing clients remain compatible when optional inferred fields are absent.

### 5. Duplicate Suppression

The canonical RDF relationship is the duplicate key.

When a fact is both asserted and entailed:

- the asserted entry or edge is returned as primary;
- no second inferred row, hierarchy placement, or edge is rendered;
- optional detail metadata may state that the asserted fact is also entailed;
- asserted styling is retained.

Duplicate suppression occurs in Kotlin before pagination and limit accounting so duplicates do not consume inferred capacity or appear on later pages.

### 6. Project Outline And Entity Details

With the filter enabled, inferred facts appear in existing semantic locations rather than in a separate inferred tree:

- class superclasses and subclasses;
- class hierarchy placements;
- individual types;
- class membership lists where already present;
- object-property usage;
- incoming and outgoing object-property assertions;
- related subject/object fields already supported by entity details.

Every inferred entry:

- has a light-blue background or border;
- includes visible `Inferred` text or badge;
- exposes an accessible name containing `Inferred`;
- retains normal keyboard navigation;
- shows provenance metadata on selection or hover where the existing component supports it.

An inferred class parent produces an additional child placement beneath that parent. It does not move or replace the class's asserted placement. An inferred-only placement:

- is marked `Inferred`;
- uses the same entity identity and opens the same entity;
- is deduplicated per parent;
- is cycle-guarded;
- is ordered after asserted children using the existing deterministic label/IRI ordering;
- does not recursively synthesize a second inferred-only tree beyond facts returned within the bounded response.

The outline does not invent groups or inferred property schemas.

### 7. Fact Details And Explanation

When an existing detail surface can represent fact provenance, an inferred entry shows:

- `Origin: Inferred`;
- inference type;
- reasoning job/result reference;
- current graph fingerprint abbreviation;
- freshness state.

`View reasoning explanation` may appear only when the existing explanation service already supports that exact retained fact and the current user can access it. Otherwise no new explanation workflow is introduced.

### 8. Ontology Map Overlay

When the filter is enabled, the ontology map may include:

- inferred `SubclassOf` edges;
- inferred `Type` edges;
- inferred `ObjectAssertion` edges.

It does not add inferred `Domain` or `Range` edges.

Inferred edges use:

- light-blue color;
- dashed stroke;
- an `Inferred` label, badge, tooltip, or accessible description in addition to color;
- lower default emphasis than asserted edges;
- a clear hover/focus treatment.

The map legend identifies `Asserted relationship` and `Inferred relationship`. Inferred edges may participate in existing node selection/focus highlighting, but do not introduce a new edge-details card or editing action.

Asserted subclass edges alone determine the hierarchy-first tree. Inferred edges are overlay cross-links and do not affect root selection, tree depth, sibling ordering, connected-component placement, or automatic layout scoring. Inferred-only endpoint nodes are inserted near an already loaded endpoint using the existing bounded expansion behavior; they do not trigger a full layout rerun.

Enabling or disabling inferred facts preserves:

- current pan and zoom;
- manually moved node positions;
- expanded and collapsed branches;
- selected entity when it remains loaded;
- Focus or Full map mode.

### 9. Limits And Pagination

Phase 10.5 keeps existing bounds and applies them to asserted and inferred data together:

- initial map page: at most 75 nodes and 150 edges;
- neighborhood expansion: at most 50 nodes and 100 edges;
- open map tab aggregate: at most 300 nodes and 600 edges;
- Project Outline/entity read page: the existing 100-item page limit;
- inferred facts per entity response: at most 100 after asserted-duplicate suppression.

Asserted content has priority within a combined response. Remaining capacity is filled with inferred facts in deterministic fact-family, subject-label, predicate-label, object-label, and canonical-IRI order.

If inferred content exceeds remaining capacity:

- the response reports truncation and available counts;
- existing continuation/load-more behavior is reused;
- no entire reasoning result is sent to the browser;
- inferred-only endpoints count against the same node limit;
- asserted facts are never displaced merely because the inferred toggle is on.

### 10. Freshness And Invalidation

The server rechecks the applied graph fingerprint when serving every inferred overlay page or continuation.

If the graph changes:

- the prior inferred overlay becomes stale;
- inferred rows and edges are removed or withheld immediately;
- asserted content remains visible;
- continuations bound to the old fingerprint fail with the existing stale-response behavior;
- the UI shows `Reasoning results are stale` and links to Reasoning;
- React clears cached inferred pages for the project.

Failed, incomplete, cancelled, or timed-out reasoning results are never treated as partial current knowledge.

### 11. Authorization And Isolation

Every inferred overlay request uses the existing development identity and approved-project boundary. The server verifies:

- current user identity;
- project registration and browse permission;
- requested source scope;
- access to the retained reasoning result;
- graph fingerprint and continuation ownership.

The response must not expose facts, job details, source paths, raw source text, secrets, or external/imported entities outside the user's existing read scope.

### 12. Read-Only Guarantee

Filter and overlay interactions use GET/read queries only, apart from ordinary navigation state.

They must not call:

- shared staging routes;
- materialization routes;
- proposal creation, approval, rejection, or apply routes;
- ontology edit translation;
- source-writing services.

The existing Reasoning materialization feature remains available only in its existing workflow and is not invoked by this filter.

## Inputs

- Current authenticated development user.
- Registered project ID.
- Selected source IDs and current Project Outline filters.
- `includeInferred` boolean, default `false`.
- Current applied-graph fingerprint or server-bound continuation.
- Existing entity, outline, graph-initial, or graph-neighborhood query.
- A retained semantic job/result reference selected by the server.

The browser never supplies inferred triples, provenance, or arbitrary reasoning job ownership claims.

## Outputs

- Filter availability/freshness metadata.
- Existing asserted Explore data.
- Optional bounded provenance-aware inferred entries.
- Optional inferred hierarchy placements.
- Optional bounded inferred graph nodes and edges.
- Stable fact identities, reasoning reference, fingerprint, origin, and inference type.
- Deterministic counts, truncation state, and opaque continuation.
- Safe unavailable, stale, loading, or failed messages.
- Updated map legend and accessible inferred styling.

No output is a staged edit, proposal, applied assertion, or source mutation.

## Validation Behavior

Kotlin and the server deterministically validate:

- project and user access;
- applied reasoning scope;
- completed and complete result state;
- graph fingerprint equality;
- source-scope visibility;
- supported inference family;
- named-IRI terms;
- object-property predicate validity;
- canonical identity;
- asserted/inferred duplicate suppression;
- deterministic ordering;
- per-response and aggregate bounds;
- continuation signature and freshness.

Malformed or unsupported facts are omitted from the Explore overlay and counted in safe diagnostics. They remain visible through existing Reasoning behavior when that behavior already supports them.

React validates only transport shape and interaction state. It never validates ontology meaning.

## Error Behavior

- No reasoning result: show `Run reasoning to view inferred facts` and a Reasoning navigation action.
- Loading result: keep asserted content usable and show bounded loading feedback.
- Stale result: hide inferred content and show `Reasoning results are stale`.
- Failed/cancelled/incomplete result: hide inferred content and show a safe status message.
- Unauthorized project/source/result: return the existing safe authorization response without existence leakage.
- Malformed or mismatched continuation: return the existing invalid/stale continuation response.
- Truncated result: retain loaded facts and show deterministic load-more/count feedback.
- Network failure: retain asserted content, hide inferred overlay, and allow the existing retry behavior.
- Unsupported fact: omit it from Explore and do not reinterpret it in React.

Errors never fall back to stale inferred facts and never mutate staged, proposal, or ontology state.

## Test Cases

### Core And Semantic Service

- Extract each supported inferred fact family from a complete current result.
- Reject proposal-scoped, failed, cancelled, incomplete, and stale results.
- Reject blank-node, literal-object, and invalid object-property facts.
- Produce stable identities and deterministic order across repeated runs.
- Suppress an inferred fact when the canonical triple is asserted.
- Prefer asserted provenance when the same fact has both origins.
- Keep inferred hierarchy placement additive and cycle-safe.
- Enforce entity, page, and graph bounds after deduplication.
- Exclude facts outside selected source/read scope.

### Server Contracts And Routes

- `includeInferred=false` preserves existing response shape and behavior.
- A current retained applied result returns provenance-aware inferred entries.
- No result, loading, stale, failed, cancelled, and incomplete states map to safe explicit metadata.
- Cross-project and unauthorized result access is rejected.
- Old-fingerprint continuations are rejected after a graph change.
- Asserted results receive priority when combined limits are reached.
- The overlay route never invokes staging, materialization, proposal, or apply services.

### Project Outline And Entity Details

- The toggle is off on first project open.
- The toggle persists across Explore tab changes and resets on project change/reload.
- Reset filters turns inferred facts off.
- Inferred superclass/subclass, individual type, and object assertion appear in their corresponding fields.
- Inferred multi-parent placement does not move asserted placement.
- Inferred entries use light blue plus visible and accessible `Inferred` text.
- Asserted/inferred duplicates render once with asserted styling.
- Unavailable and stale messages navigate to Reasoning without starting a job.
- Keyboard navigation reaches inferred entries.

### Ontology Map

- Supported inferred edges render dashed, light blue, and explicitly inferred.
- The legend distinguishes asserted and inferred relationships.
- An asserted equivalent suppresses the inferred edge.
- Inferred edges do not affect primary hierarchy layout.
- Enabling/disabling inferred facts preserves pan, zoom, positions, expansion, selection, and mode.
- Inferred-only endpoints remain bounded and are inserted near a loaded endpoint.
- Combined asserted/inferred graph limits and load-more behavior hold.
- Focus mode includes directly related inferred edges when enabled.
- No inferred literal, domain, or range edge appears.

### Regression And Scale

- Existing Explore, Reasoning, materialization, staging, proposal, CLI, and VS Code tests remain green.
- Existing asserted-only map snapshots and interactions remain unchanged when off.
- Small, medium, 500-entity, 1,000-entity, and dense-inference fixtures remain bounded and deterministic.
- Accessibility tests cover non-color origin labels, keyboard behavior, contrast, and reduced motion.
- End-to-end tests cover current, unavailable, stale, and duplicate-suppressed flows.

## Acceptance Criteria

1. The Project Outline filter popover includes `Inferred facts`.
2. The toggle is off by default and reset by `Reset filters`.
3. One project-scoped temporary toggle controls Project Outline, entity details, and map.
4. Enabling it shows supported facts only from a current, complete, applied-graph reasoning result.
5. Inferred facts appear in the same fields as equivalent asserted facts.
6. Inferred entries are light blue and explicitly identified as `Inferred`.
7. Inferred hierarchy placements are additive and do not replace asserted placement.
8. Canonical asserted/inferred duplicates render once as asserted.
9. The map shows dashed, light-blue, explicitly labeled inferred subclass, type, and object-assertion edges.
10. Asserted hierarchy alone controls primary map layout.
11. Filter changes preserve temporary map interaction state.
12. Unavailable, loading, stale, failed, and incomplete states are safe and understandable.
13. Existing graph and Explore bounds apply to asserted and inferred data together.
14. Cross-project, cross-user, and out-of-scope source data is not exposed.
15. The feature remains read-only and makes no staging, proposal, apply, or source-write call.
16. Kotlin remains authoritative for inferred meaning, provenance, deduplication, freshness, ordering, and limits.
17. Existing behavior is unchanged when the toggle is off.
18. Deterministic unit, integration, frontend, accessibility, scale, and end-to-end verification passes.

## Boundary Check

- Phase fit: this is the exact read-only visibility feature defined by the Phase 10.5 scope.
- Current non-goals: no materialization, editing, AI generation, new inference rules, persistence, CLI work, or VS Code work is added.
- Infrastructure: existing reasoning results, semantic services, Ktor contracts, React Query, Project Outline filters, and Phase 9 map services are reused.
- Module ownership: Kotlin owns semantics and provenance; Ktor owns authorized bounded reads; React owns display and temporary filter/map state.
- Standards: existing OWL API/HermiT reasoning output and RDF contracts are reused; no RDF, OWL, or SHACL implementation is invented.

## Open Questions

None blocking.

The following decisions are intentionally fixed for Phase 10.5:

- stale inferred facts are hidden, not displayed under a stale overlay;
- enabling the filter never starts reasoning automatically;
- filter state is temporary per open project and is not persisted across reload;
- inferred multi-parent placements are additive marked rows;
- inferred domains and ranges are excluded;
- map edges reuse current interactions and do not gain a new details/editing workflow;
- reasoning explanations are linked only when the existing explanation flow supports the exact fact;
- inferred-only endpoints load within existing graph limits and continuations, never through an unbounded fetch.
