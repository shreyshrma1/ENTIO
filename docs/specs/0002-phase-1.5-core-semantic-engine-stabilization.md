# Feature Spec: Phase 1.5 Core Semantic Engine Stabilization

## Status

Implemented

## Problem

Phase 1 created the first Kotlin/JVM Core Semantic Engine foundation, but two important stabilization gaps remain before later phases build on it.

First, project loading is not exposed through one reusable engine API. The current behavior is split across config loading, source resolution, ontology parsing, symbol extraction, validation orchestration, and CLI-specific project reading. This means consumers such as the CLI can load useful project state, but there is no canonical `ProjectLoader` that constructs an `EntioProject`.

Second, the Phase 1 RDF graph model is lossy. `GraphTriple` currently stores subject and predicate as IRIs, but collapses every object into a single `String` value. That loses RDF term distinctions such as IRI resources, blank nodes, plain literals, datatyped literals, and language-tagged literals. Later ontology mutation, validation, Schema RAG, document ingestion, and agent workflows should not depend on that lossy representation.

Phase 1.5 should stabilize these contracts while preserving the small backend/core-engine scope.

## Goals

- Make Phase 1 project loading reusable through a public `ProjectLoader` API in `semantic-engine`.
- Construct and return a populated `EntioProject` aggregate.
- Move project-loading orchestration out of CLI-specific code.
- Preserve RDF term fidelity in Entio-owned core types.
- Distinguish RDF IRI resources, blank-node resources, and literals.
- Preserve literal datatype IRIs and language tags.
- Prevent literals from being represented as RDF subjects through the type model.
- Update parsing, symbol extraction, validation, graph diffing, formatting, and CLI behavior to use the corrected graph model.
- Preserve existing Phase 1 behavior where compatible.
- Keep deterministic output and tests.
- Keep Jena usage contained inside `semantic-engine`.

## Non-Goals

- No ontology mutation or persistence of edited graphs.
- No add, remove, rename, or delete operations for ontology concepts or triples.
- No change proposal application, approval, undo, redo, rollback, or version history.
- No OWL reasoning or full OWL axiom interpretation.
- No SHACL validation engine or SHACL authoring environment.
- No Schema RAG.
- No document ingestion.
- No LLM integration or autonomous agents.
- No entity resolution.
- No human review UI or approval workflow.
- No VS Code extension.
- No web app.
- No API server, server mode, or watch mode.
- No named graph support unless explicitly approved by a later spec.
- No full source-text-preserving Turtle round trips.
- No custom RDF, OWL, SHACL, or Turtle framework.

## Proposed Behavior

Phase 1.5 should leave Entio with a reusable, semantically safer core loading path.

The semantic engine should expose one public project-loading entry point:

```kotlin
fun loadProject(projectRoot: Path): EntioResult<EntioProject>
```

The loader should:

- Load project configuration from `entio.yaml`.
- Resolve ontology source references.
- Parse each resolved ontology source.
- Extract symbols from parsed ontologies.
- Assemble source-specific loaded ontologies.
- Assemble a deterministic combined symbol list.
- Assemble a deterministic combined graph state.
- Return a populated `EntioProject`.
- Return structured failures when config loading, source resolution, parsing, or symbol extraction fails.

The CLI should delegate project loading to this reusable API rather than owning project-loading orchestration.

The graph model should preserve RDF term shape:

- Subjects should be RDF resources, not literals.
- Predicates should be IRI resources.
- Objects should be RDF terms.
- RDF terms should include IRI resources, blank-node resources, and literals.
- Literals should preserve lexical form plus optional datatype IRI and optional language tag.

Existing commands should continue to work:

```text
entio validate <project-root>
entio symbols <project-root>
entio diff <before-project-root> <after-project-root>
```

Output may change only where needed to represent RDF terms correctly and deterministically.

## Inputs And Outputs

### Inputs

Phase 1.5 uses the same local project input shape as Phase 1:

```text
project-root/
  entio.yaml
  ontology/
    simple.ttl
```

Minimal config remains:

```yaml
name: simple-ontology
ontologySources:
  - id: simple
    path: ontology/simple.ttl
    format: turtle
```

Phase 1.5 should continue to support Turtle ontology sources.

### Outputs

Project loading should return:

- `EntioResult.Success<EntioProject>` when loading succeeds.
- `EntioResult.Failure` with structured issues when loading fails.

`EntioProject` should contain:

- loaded project configuration,
- resolved ontology sources,
- source-specific loaded ontologies,
- extracted symbols,
- combined graph state.

Parsed graphs should contain triples that preserve:

- IRI subjects,
- blank-node subjects,
- IRI predicates,
- IRI objects,
- blank-node objects,
- plain literals,
- datatyped literals,
- language-tagged literals.

CLI commands should continue to produce readable text output and stable exit codes.

## Core Data Contracts

Phase 1.5 should introduce or refine these core contracts:

- `ProjectLoader` in `semantic-engine`.
- `EntioProject` as the canonical loaded-project aggregate.
- `RdfResource` for RDF subjects and other non-literal resources.
- `RdfTerm` for RDF objects.
- IRI-backed RDF resources.
- Blank-node RDF resources.
- Literal RDF terms with lexical form, optional datatype IRI, and optional language tag.
- Updated `GraphTriple` with:
  - subject: `RdfResource`,
  - predicate: IRI-backed resource or `Iri`,
  - object: `RdfTerm`.
- Updated `GraphState` over the corrected triple model.

Exact class, sealed-interface, and enum names may be refined during ExecPlan creation, but the contracts must preserve the distinctions above.

## Validation Behavior

Phase 1.5 should preserve Phase 1 deterministic validation behavior:

- Project root exists and is a directory.
- `entio.yaml` exists.
- `entio.yaml` is valid YAML.
- Required config fields are present.
- Ontology source IDs are unique.
- Ontology paths are relative, normalized, and remain under the project root.
- Referenced ontology files exist.
- Ontology formats are supported.
- Turtle files parse successfully.
- Loaded symbols can be extracted without crashing.

Additional validation expectations:

- Project loading failures should be structured and deterministic.
- RDF parsing failures should remain attached to the relevant ontology source ID.
- Symbol extraction should work with the corrected RDF term model.
- Validation should not require callers to duplicate the loading sequence.

## Error Behavior

Expected project problems should return `EntioResult.Failure` or `ValidationReport` values, not unstructured crashes.

Project loading should fail deterministically when:

- config loading fails,
- ontology source resolution fails,
- one or more ontology sources fail to parse,
- symbol extraction fails unexpectedly.

RDF term conversion should not silently collapse term types. If a parsed RDF node cannot be represented by the Phase 1.5 term model, the failure should be structured and test-covered.

## Test Cases

- `ProjectLoader` loads `examples/simple-ontology` successfully.
- `ProjectLoader` returns an `EntioProject` containing config, resolved sources, loaded ontologies, symbols, and combined graph state.
- `ProjectLoader` returns a structured failure when config loading fails.
- `ProjectLoader` returns a structured failure when source resolution fails.
- `ProjectLoader` returns a structured failure when Turtle parsing fails.
- Multiple ontology sources load in deterministic order.
- Duplicate triples across ontology sources collapse in the combined graph without removing source-specific ontology graphs.
- Turtle parsing preserves IRI subjects and IRI objects.
- Turtle parsing preserves blank-node subjects.
- Turtle parsing preserves blank-node objects.
- Turtle parsing preserves plain literals.
- Turtle parsing preserves datatyped literals.
- Turtle parsing preserves language-tagged literals.
- The Entio type model prevents literal subjects.
- Symbol extraction still identifies classes, properties, individuals, and shapes under the corrected term model.
- Validation reports remain deterministic.
- Graph diffs still report added and removed triples under the corrected term model.
- Graph diffs still detect label changes under the corrected term model.
- Diff formatting remains stable and readable.
- CLI `validate` continues to return zero for valid projects and non-zero for invalid projects.
- CLI `symbols` delegates to `ProjectLoader` and keeps stable output.
- CLI `diff` delegates project loading to `ProjectLoader` and keeps deterministic output.
- Existing Phase 1 tests are updated as needed and continue to pass.

## Acceptance Criteria

- A public `ProjectLoader` API exists in `semantic-engine`.
- `ProjectLoader.loadProject(projectRoot)` returns `EntioResult<EntioProject>`.
- Successful loading returns an `EntioProject` containing config, resolved sources, loaded ontologies, extracted symbols, and combined graph state.
- CLI project-loading orchestration is removed or reduced to delegation to `ProjectLoader`.
- Core graph types preserve RDF resource and term distinctions.
- Literal datatype IRIs and language tags are preserved.
- Blank nodes can participate in parsed graphs.
- Literals cannot be modeled as triple subjects.
- Existing Phase 1 CLI commands still work.
- Existing validation, symbol extraction, and graph diff behavior remains deterministic.
- New tests cover project aggregation and RDF term fidelity.
- `./gradlew test` passes.
- `./gradlew build` passes.
- `./gradlew check` passes.
- No Phase 2 or later product behavior is introduced.

## Open Questions

- What exact names should the RDF term sealed interfaces and data classes use?
- Should predicates remain represented by `Iri`, or should they use a more specific IRI resource type?
- How should blank-node identifiers be formatted in user-facing CLI and diff output while making clear they are not durable semantic identities?
- Should `EntioProject` include both source-specific symbols and a combined symbol list, or only the combined list currently defined in Phase 1?
- Should project loading collect all parse failures across sources or stop at the first failure?
- Should `ProjectLoader` be used by `ProjectValidator`, or should validation continue to orchestrate lower-level services independently to preserve validation-specific issue reporting?
