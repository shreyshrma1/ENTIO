# Feature Spec: Phase 2.5+ Label-First Editing, Deletion, And Staged Change Sets

## Status

Draft

## Problem

Phase 2.5 completed the basic user-facing ontology editing surface and reused the existing Kotlin proposal workflow for preview, validation, semantic diffing, approval, atomic application, reload, and rollback. The workbench can now request several typed edits, but users still have to identify existing entities primarily through technical IRIs, cannot safely delete supported ontology elements, and cannot prepare several edits for one combined review.

This makes ordinary ontology maintenance harder than it needs to be and leaves the review workflow fragmented. Entio needs a label-first interaction model, deterministic identifiers for newly created local elements, explicit deletion dependency handling, and an in-memory staged change set that can be reviewed and applied as one atomic proposal.

## Goals

- Let users select and edit existing ontology elements primarily by human-readable label.
- Keep full IRIs visible as technical identity and disambiguation information.
- Generate deterministic unique IRIs for new classes, properties, and individuals.
- Detect identifier collisions and duplicate creation before source mutation.
- Support deletion proposals for classes, object properties, datatype properties, and individuals.
- Show explicit dependent references before deletion and block ambiguous cascading deletion by default.
- Allow the workbench to stage multiple individually previewed edits in one editing session.
- Allow a staged edit to be edited, re-previewed, cancelled, or removed before application.
- Produce one deterministic combined preview, semantic diff, validation report, round-trip result, and atomic save for a staged change set.
- Preserve the Kotlin semantic engine as the only owner of label resolution, IRI generation, graph changes, validation, serialization, and persistence.
- Extend the machine-readable CLI boundary so the workbench can use the same contracts.

## Non-Goals

- Schema RAG, AI-generated ontology edits, document ingestion, document-to-knowledge-graph conversion, entity resolution, or autonomous agents.
- Full OWL reasoning, full SHACL authoring or validation, or inferred dependency analysis.
- Equivalent-class, disjoint-class, inverse-property, property-chain, cardinality, or property-characteristic editing.
- Full Protégé parity or a drag-and-drop graph editor.
- Long-term staged-change persistence, multi-user review queues, authentication, authorization, or a server/database.
- Source-text-preserving Turtle editing.
- A separate persistence or editing pathway in TypeScript.
- Actual Git staging, commits, pushes, branches, pull requests, or version-history behavior inside Entio. The review flow may remain git-like by analogy only.
- Random identifiers, silent overwrites, or unrestricted manual IRI entry as the normal workflow.
- Changes to the existing project/module layout unless a later approved implementation slice proves that a focused change is necessary.

The current phase-level non-goals in `AGENTS.md` remain in force.

## Proposed Behavior

### Label-First Selection And Details

Existing entities should be presented using a preferred human-readable label, entity kind, and source ontology where needed. The full IRI remains visible in the selected entity details and in secondary preview information.

Label resolution must be deterministic:

1. Trim the requested label.
2. Match preferred labels from the current loaded project using the supported label predicate.
3. Apply explicit kind and source filters when provided.
4. Return one entity only when the remaining match is unique.
5. Return a structured ambiguity issue with candidate details when more than one match remains.
6. Return a structured missing-entity issue when no match remains.

The resolver must not perform fuzzy matching, external lookup, or inference. Machine-readable requests may continue to provide IRIs directly for compatibility, but ordinary workbench forms should use label-backed controls.

The details panel should show, where available:

- Preferred label.
- Full IRI.
- Entity kind.
- Source ontology.
- Direct superclass, domain, range, or type statements.
- Supported incoming and outgoing statements involving the entity.

### Project Namespace And Deterministic IRI Generation

New local elements require an approved project namespace. The project configuration should support a top-level `iriNamespace` value:

```yaml
name: simple-ontology
iriNamespace: https://example.com/entio/simple#
ontologySources:
  - id: simple
    path: ontology/simple.ttl
    format: turtle
```

The namespace must be an absolute IRI and should end in `#` or `/` so local-name concatenation is unambiguous. Existing projects without `iriNamespace` remain loadable for read-only operations and explicit-IRI edits, but label-first creation must fail deterministically with a missing-namespace validation issue until the configuration is supplied. A source-specific namespace override is out of scope for this phase.

The initial local-name convention is:

- Trim surrounding whitespace and collapse internal whitespace.
- Split on unsupported local-name characters and discard empty segments.
- For classes and individuals, capitalize each segment and concatenate them as UpperCamelCase.
- For object and datatype properties, capitalize each segment after the first and concatenate them as lowerCamelCase.
- If the first usable segment begins with a digit, prefix `Item` for classes/individuals or `item` for properties.
- Reject labels that produce no usable segment.
- Use the same normalized output for the same label, kind, and normalization version.

The generated candidate is:

```text
configured iriNamespace + normalized local name
```

If the candidate IRI already exists, Entio must never overwrite it. The default behavior is to reject creation when the current graph contains an entity with the same normalized label and kind. An explicit distinct-entity request may use the first available deterministic suffix:

```text
normalized local name
normalized local name__2
normalized local name__3
```

The suffix search examines the current project graph in ascending numeric order. It is deterministic, does not use randomness or timestamps, and must recheck the graph immediately before proposal preparation. The generated IRI must be shown in the preview, staged-change details, selected-item details, and relevant validation messages.

The normalization convention should be versioned in the generated request/result metadata so a later convention change cannot silently reinterpret an existing staged request.

### Deletion Proposals

Supported deletion kinds are classes, object properties, datatype properties, and individuals. Deletion follows the existing proposal lifecycle and never writes the source immediately.

For a selected entity, the Kotlin engine should find explicit graph statements that define or reference it. Direct deletion includes the selected entity's supported type declarations, labels and in-scope annotations, direct hierarchy/domain/range statements where it is the subject, and supported assertions where it is the subject. Statements that refer to it from another subject are reported as dependent references.

The safe default is to block staging while unresolved dependent references remain. The user may explicitly select supported dependent statements for removal in the same staged change set. Entio must show the selected removals and blockers; it must not silently cascade through the graph. Replacing a reference with another entity is a separate edit and is not implicit in deletion.

Deletion validation must reject missing entities, wrong-source requests, unresolved dependencies, stale baselines, and combined graphs that fail deterministic validation. The analysis operates only on explicit current graph statements and does not attempt full OWL dependency reasoning.

### Workbench Deletion Controls

Deletion must be an explicit workbench action on the selected symbol, rather than an implicit consequence of editing another form. From the symbol details view, the user can choose `Delete` for a supported class, object property, datatype property, or individual. The workbench opens a deletion review containing:

- The target label, kind, source, and full IRI in the technical details.
- Direct definition statements that are always part of deleting the selected entity and are not individually optional.
- Each explicit dependent statement as a separately identifiable selectable item, rendered label-first where labels are available.
- A readable fallback name for RDF, RDFS, OWL, and XML Schema vocabulary terms instead of exposing their namespace IRIs in the primary review text.

The `Preview deletion` action remains disabled while required dependent statements are unresolved or while the dependency inspection has not completed. The user must explicitly select every dependent statement to be removed, or cancel the deletion. Selecting a dependent statement does not imply selection of any other statement. The workbench sends the target source and IRI together with stable dependent-statement identities to the Kotlin boundary, which returns the deletion proposal through the existing staged and combined proposal lifecycle.

After a successful deletion preview, the deletion appears in the staged list and participates in combined preview, approval, rejection, application, refresh, and rollback exactly like other edits. Rejection, validation failure, stale state, or apply failure leaves the source unchanged. On successful application, the workbench reloads the project and clears or refreshes the affected staged state.

### Staged Change Sessions

The VS Code workbench maintains an ordered staged-change list in memory for the active session. Each entry contains a stable session identifier, display summary, edit kind, target source, normalized values, generated or resolved IRIs, individual validation status, and the underlying typed edit/request needed to recreate it.

The expected interaction is:

1. The user fills an edit form.
2. `Preview Change` submits one edit to the Kotlin boundary.
3. Entio translates and validates that edit without changing source files.
4. A successful preview is added to the staged list and the form is cleared.
5. A failed preview leaves the form populated and does not stage the invalid edit.
6. The user may prepare additional edits, edit an existing staged item, or remove it.

Editing a staged item temporarily removes it from combined evaluation or marks it as being edited, restores its form values, and returns the revised version only after a successful re-preview. Cancelling restores the prior staged version. Removing an entry changes only the in-memory session list.

The CLI should support the same model through one structured request containing an ordered list of edits. CLI invocations remain stateless; this phase does not persist staged sessions between processes. The workbench is responsible for its active session state and the Kotlin boundary is responsible for normalizing and verifying the submitted ordered list.

### Combined Preview And Atomic Application

Before final approval, Entio must:

1. Reload or confirm the current project baseline.
2. Normalize staged edits and preserve their deterministic order.
3. Detect duplicate and conflicting edits.
4. Apply the complete ordered change set to an in-memory graph.
5. Generate one combined semantic diff.
6. Validate the final proposed graph and report contributing staged entries.
7. Serialize the preview to temporary Turtle.
8. Reparse it and verify semantic equivalence.
9. Present the complete result, affected source files, and approval state.
10. Apply the complete approved set atomically through the existing applier.
11. Reload the project and refresh the workbench after success.

Conflicts include creating and deleting the same entity, duplicate generated IRIs, deleting an entity required by another staged edit, referencing an entity removed earlier in the order, incompatible property value/range edits, and changes that become empty after normalization. The order must be visible when it affects conflict or validation results.

No source may change unless the combined semantic diff exists, the final graph is valid, Turtle round-trip verification succeeds, and the complete set is explicitly approved. Rejection, failed validation, stale baselines, serialization failure, and post-save verification failure leave source files unchanged. A save or verification failure must return the existing structured rollback result, and no partial subset may remain applied.

After successful application, the source reloads, the workbench refreshes, and the staged list is cleared. The existing changed-source opening action remains available.

### CLI Boundary

The existing `proposal-preview`, `proposal-validate`, `proposal-diff`, `proposal-apply`, and `proposal-reject` commands remain backward compatible for single edits. Phase 2.5+ adds a structured request-file boundary for heterogeneous staged edits rather than making command flags represent an unbounded list.

The structured request contains:

- Project root and target source information.
- Ordered edit entries.
- Edit kind and form values.
- Resolved existing IRIs.
- Generated IRIs and normalization metadata.
- Explicit deletion dependency selections where applicable.
- Baseline information and proposal metadata.

The CLI should also expose machine-readable operations for label resolution, deterministic IRI generation, and deletion dependency inspection. The semantic work remains in Kotlin modules; CLI code only parses requests, invokes shared services, and formats structured results.

Responses should include normalized edits, identifiers, individual and combined validation results, semantic diff, target and affected sources, baseline/equivalence information, and apply/reject/rollback status where relevant.

### VS Code Boundary

The extension should provide label-backed selectors, generated-IRI display, supported delete actions from symbol details, deletion review with explicit dependent-statement selection, readable vocabulary labels, staged-list controls, form clearing after successful preview, staged-edit restore/edit/cancel behavior, combined preview, approval, rejection, application, and refresh. A deletion preview must not be available until the required dependency selections are explicit.

TypeScript owns form state, active staged-list state, rendering, CLI invocation, and result presentation. Kotlin owns label resolution, IRI generation, collision checks, deletion analysis, graph changes, conflict detection, combined preview, validation, round-trip verification, and atomic application. The extension must not construct RDF, generate final identifiers, resolve labels independently, combine graph changes, write Turtle, or bypass approval.

## Inputs And Outputs

### Inputs

- A local Entio project containing `entio.yaml` and one or more configured ontology sources.
- An optional project-level `iriNamespace` for generated identifiers.
- A target ontology source.
- A label-backed selection or explicit IRI for compatibility.
- An edit kind and edit-specific fields.
- Existing classes, properties, individuals, and resources when referenced.
- An ordered list of staged edits for combined CLI preview/application.
- A deletion target and explicit dependent-statement selections when the user requests deletion.
- Explicit approval or rejection.

### Core Contracts

The implementation should introduce or extend Entio-specific contracts for:

- Namespace configuration and normalization metadata.
- Label/entity selectors and resolved entity candidates.
- Generated IRI results and collision decisions.
- Deletion dependency summaries and explicit dependent-triple selections.
- Staged changes, staged change sets, conflict results, and combined proposal metadata.
- Per-entry and combined validation attribution.

These contracts must use Entio data types and must not expose Apache Jena types outside the semantic-engine boundary.

### Outputs

- Label resolution results with zero, one, or multiple candidates.
- Deterministically generated IRIs and collision status.
- Deletion previews and dependency summaries.
- Normalized staged edits with deterministic order.
- Combined preview graph and one semantic diff.
- Per-edit and combined deterministic validation reports.
- Turtle semantic-equivalence results.
- Affected source files and applied/rejected/stale/failed/rolled-back outcomes.
- Refreshed workbench state and cleared staged list after successful application.

## Validation And Error Handling

Validation must be deterministic, stable, and based on explicit graph facts:

- Missing or invalid labels.
- Missing or invalid namespace configuration.
- Invalid generated local names.
- Generated IRI collisions and duplicate normalized entities.
- Ambiguous label selections.
- Entity kind or source mismatches.
- Missing deletion targets.
- Unresolved deletion dependencies.
- Duplicate staged changes.
- Conflicting staged changes and references to entities removed by another staged edit.
- Empty normalized change sets.
- Stale project or source baselines.
- Unsupported dependent-triple removals.
- Final Turtle round-trip non-equivalence.

CLI failures return a nonzero exit code with structured JSON. Workbench failures remain visible near the relevant form or staged entry and retain the invalid form data for correction. No error path may write source files before combined preview, validation, round-trip verification, and explicit approval have succeeded.

## Test Cases

### Label-First And IRI Generation

- Resolve one unique label to the correct IRI.
- Reject ambiguous labels and return deterministic candidates.
- Show full IRI, kind, and source in entity details.
- Normalize class, individual, and property labels deterministically.
- Generate the same IRI for repeated valid input.
- Reject missing namespace configuration.
- Reject collisions or apply the explicit deterministic suffix strategy.
- Preserve generated-IRI metadata through preview and staged results.

### Deletion

- Delete a class with no explicit references.
- Delete an object property with no assertions.
- Delete a datatype property.
- Delete an individual.
- Report incoming and outgoing dependent references.
- Block unresolved deletion until every required dependent statement is explicitly selected or the user cancels.
- Remove explicitly approved dependent statements in the same change set.
- Reject missing targets, wrong sources, stale baselines, and invalid combined graphs.
- Preserve source files on rejection or failed application.

### Staging And Combined Application

- Stage at least two independent edits without changing source files.
- Clear the form after successful preview and retain it after failure.
- Edit, re-preview, replace, cancel, and remove staged entries.
- Detect duplicate and conflicting staged changes.
- Produce one combined semantic diff and validation report.
- Attribute combined failures to contributing staged entries.
- Apply all approved changes atomically.
- Preserve every staged entry when final validation fails.
- Clear the staged list and refresh after success.

### Integration

- Parse and emit a structured multi-edit CLI request and response.
- Preserve single-edit CLI backward compatibility.
- Exercise workbench label selectors, generated-IRI details, symbol-details Delete actions, deletion dependency checkboxes, blocked and successful deletion preview, staged-list actions, combined preview, approval, rejection, application, and refresh.
- Verify round-trip persistence, stale proposal blocking, and rollback after failed post-save verification.
- Use temporary copies of `examples/simple-ontology` or temporary fixtures; never modify committed examples during tests.

## Acceptance Criteria

Phase 2.5+ is accepted when:

- Existing ontology items can be selected primarily by label and their full IRIs remain visible.
- New supported classes, properties, and individuals receive deterministic unique IRIs without manual IRI entry.
- Collisions and ambiguous labels are rejected or handled through the approved deterministic strategy.
- Supported deletions produce reviewable proposals and expose dependent references before staging.
- The workbench provides a Delete action for supported symbols, renders direct definitions separately from dependent statements, and requires explicit dependent-statement selection before Preview deletion is enabled.
- Ambiguous deletion cascades are blocked rather than silently applied.
- Multiple edits can be previewed and staged without source mutation.
- Staged edits can be edited, re-previewed, cancelled, or removed.
- One combined preview, semantic diff, validation result, round-trip result, and approval is available for the complete staged set.
- Rejection and failed validation leave every source unchanged.
- Approved staged edits apply atomically and survive reload.
- Save/verification failure restores the prior source and exposes rollback status.
- The CLI and VS Code extension consume the same Kotlin-owned semantic behavior.
- Focused tests and copied-fixture end-to-end tests cover success and failure paths.
- Existing Gradle and extension verification commands pass.
- No new RDF/OWL/SHACL/Turtle framework, Git workflow, long-term persistence, or later-phase infrastructure is introduced.

## Open Questions

- Should `iriNamespace` be required for every project or only for projects that request generated identifiers?
- Should the explicit distinct-label collision request be exposed in the workbench, or remain an advanced CLI/API option?
- Should the normalization convention support Unicode letters and digits in the first implementation, or restrict local names to ASCII?
- Resolved design decision: one deletion proposal supports explicit removal of the selected dependent statements; unselected or unresolved dependencies block preview. A separate proposal remains available for replacing a reference or making another independent edit.
- Should label resolution prefer language-tagged labels based on a project setting, or treat all matching labels equally until a language policy exists?
- Should structured requests be passed through a temporary JSON file, stdin, or both?
- Should staged entries receive durable IDs only for the active workbench session, or should the Kotlin response also include a deterministic display key?
- Should the workbench allow multiple target sources in one staged set, or require one target source per combined application?
- How should blank nodes appear in deletion dependency details without implying stable user-facing identity?
- Should explicit superclass-cycle checks be included now, or remain limited to direct self-subclass and existing deterministic checks?

## Boundary Check

This feature fits the requested next phase because it improves the usability and reviewability of the existing controlled editing workbench. It keeps project loading, RDF interpretation, graph changes, validation, semantic diffs, Turtle round-trip checks, and source mutation in the Kotlin engine. It keeps session/UI behavior in the extension and does not introduce web, database, AI, document, Git, full reasoning, or full OWL/SHACL infrastructure.
