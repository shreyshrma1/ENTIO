# ExecPlan: Phase 1.5 Core Semantic Engine Stabilization

## Status

Completed

## Source Spec

[Feature Spec: Phase 1.5 Core Semantic Engine Stabilization](../specs/0002-phase-1.5-core-semantic-engine-stabilization.md)

## Objective

Implement Phase 1.5 as a small stabilization pass over the completed Phase 1 Kotlin/JVM Core Semantic Engine.

The implementation should add a reusable `ProjectLoader`, construct and return `EntioProject`, move project-loading orchestration out of CLI-specific code, and correct the Entio-owned RDF graph model so parsed triples preserve RDF resources, blank nodes, plain literals, datatyped literals, and language-tagged literals.

The implementation must preserve existing Phase 1 behavior where compatible and must not introduce later-phase product surfaces.

## Current State

The repository already includes:

- Kotlin/JVM Gradle modules:
  - `core-types`
  - `semantic-engine`
  - `validation-engine`
  - `graph-diff`
  - `cli`
  - `shared`
- `ProjectConfigLoader` for `entio.yaml`.
- `OntologySourceResolver` for safe local source resolution.
- `OntologyParser` using Apache Jena for Turtle/RDF parsing.
- `SymbolExtractor` for basic symbol extraction.
- `ProjectValidator` for deterministic validation reports.
- `GraphDiffer` and `SemanticDiffFormatter` for basic graph diffs.
- Picocli CLI commands:
  - `validate`
  - `symbols`
  - `diff`
- `EntioProject` as a core type, but no reusable `ProjectLoader`.
- `GraphTriple` with a simplified object `String`, which does not preserve RDF term type, datatype, language, or blank-node semantics.
- `CliProjectReader`, which currently owns CLI-local project-loading orchestration for `symbols` and `diff`.

## Target State

After Phase 1.5:

- `semantic-engine` exposes a public `ProjectLoader`.
- `ProjectLoader.loadProject(projectRoot)` returns `EntioResult<EntioProject>`.
- A successful `EntioProject` includes:
  - project config,
  - resolved ontology sources,
  - source-specific loaded ontologies,
  - extracted symbols,
  - deterministic combined graph state.
- `cli` delegates project loading to `ProjectLoader` rather than owning loading orchestration.
- Core graph contracts distinguish:
  - RDF IRI resources,
  - RDF blank-node resources,
  - RDF literals,
  - literal datatype IRIs,
  - literal language tags.
- Literals cannot be represented as triple subjects.
- Apache Jena types remain contained inside `semantic-engine`.
- Validation, symbol extraction, graph diffing, formatting, and CLI behavior remain deterministic.
- Existing Phase 1 tests are updated as needed and pass.

## Implementation Sequence

### Slice 1: Additive Core RDF Term Contracts

Goal: Introduce Entio-owned RDF term contracts without breaking existing Phase 1 graph consumers.

Allowed modules:

- `core-types`.

Forbidden actions/modules:

- no edits to `semantic-engine`, `validation-engine`, `graph-diff`, `cli`, or `shared`.
- no parser, validation, diff, CLI, or project-loading behavior.
- no changes to `GraphTriple` that require downstream modules to compile against a new subject or object contract.
- no Jena, RDF4J, OWL API, or other semantic-web library types in `core-types`.
- no ontology mutation, persistence, reasoning, or UI behavior.
- no edits to build files or dependencies.

Expected changes:

- Add RDF term/resource models, such as:
  - `sealed interface RdfTerm`
  - `sealed interface RdfResource : RdfTerm`
  - IRI-backed resource representation, either by making existing `Iri` implement `RdfResource` or by adding a small dedicated IRI resource type.
  - blank-node resource data class.
  - literal data class with lexical form, datatype IRI, and language tag.
- Keep existing `GraphTriple` and `GraphState` behavior source-compatible for downstream Phase 1 modules.
- Add only additive construction or helper behavior needed to prove the new term contracts compile.

Tests:

- Construct each RDF term kind.
- Verify literals are not RDF resources through the type model.
- Verify literal datatype and language fields are preserved.
- Verify existing `GraphTriple` construction still works.

Verification:

- `./gradlew :core-types:test`
- `./gradlew check`

Stop conditions:

- Jena types leak into `core-types`.
- The new RDF resource model treats literals as resources.
- Existing Phase 1 modules fail to compile because this slice changed `GraphTriple` or `GraphState` incompatibly.
- The model adds ontology mutation or reasoning concepts.

## Rollback Notes

If a slice fails, revert that slice as a focused change.

Because Phase 1.5 changes shared graph contracts, slices that depend on `core-types` updates should not be merged until downstream modules are compatible or the slice is explicitly scoped as a breaking contract step.

If RDF term contract changes need to be backed out:

- revert the `core-types` term-model changes first,
- then revert dependent parser, symbol, validation, diff, and CLI changes,
- restore the previous `GraphTriple` contract and tests together.

Avoid partial rollback that leaves modules compiled against incompatible graph contracts.

## Risks And Assumptions

Risks:

- Updating `GraphTriple` is a cross-module breaking change.
- Blank-node handling can produce unstable output if identifiers are treated too casually.
- Literal formatting can become confusing if datatype and language metadata are omitted.
- CLI behavior may appear changed even when core semantics are more correct.
- Validation may lose useful issue reporting if it delegates too much to `ProjectLoader`.

Assumptions:

- Apache Jena remains the RDF parser for Phase 1.5.
- No new dependencies are required.
- Phase 1.5 may update existing tests to match the corrected graph model.
- The project remains local-file based.
- No version storage, ontology mutation, or human approval behavior is part of this phase.

## Boundary Check

This plan fits the active Phase 1.5 scope because it stabilizes the existing backend/core engine.

This plan avoids current non-goals:

- no VS Code extension,
- no web app,
- no document ingestion,
- no AI or autonomous agents,
- no Schema RAG,
- no entity resolution,
- no graph database integration,
- no ontology mutation,
- no custom RDF, OWL, SHACL, or Turtle framework.

The plan preserves module responsibilities:

- `core-types` owns stable Entio data contracts.
- `semantic-engine` owns project loading and Jena-backed parsing.
- `validation-engine` owns validation reports.
- `graph-diff` owns semantic graph comparison and diff formatting.
- `cli` remains a thin command-line adapter.
- `shared` remains minimal unless a concrete generic utility is explicitly justified.

The plan continues to use established semantic-web tooling rather than reinventing RDF parsing.

## Definition Of Done

Phase 1.5 is done when:

- the approved implementation slices are complete,
- `ProjectLoader` exists and returns `EntioResult<EntioProject>`,
- `EntioProject` is populated by engine behavior,
- CLI project-loading orchestration delegates to reusable engine APIs,
- RDF graph contracts preserve resources, blank nodes, literals, datatypes, and language tags,
- validation, symbol extraction, graph diffing, formatting, and CLI behavior work with the corrected graph model,
- new RDF term fidelity and project aggregation tests are present,
- existing Phase 1 behavior remains covered by tests,
- `./gradlew test` passes,
- `./gradlew build` passes,
- `./gradlew check` passes,
- no later-phase product surface or infrastructure has been introduced.

## Open Questions To Resolve Before Implementation

- Exact names for RDF term and resource data classes.
- Whether predicates should use `Iri` or a dedicated IRI resource type.
- How blank-node identifiers should be displayed in CLI and diff output.
- Whether `EntioProject` should include source-specific symbol lists in addition to a combined symbol list.
- Whether `ProjectLoader` should collect all parse failures or stop at the first failure.
- Whether `ProjectValidator` should call `ProjectLoader` or preserve its existing lower-level orchestration for more specific validation issues.
